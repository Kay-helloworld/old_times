### 一、ISO 27001 ISMS (必考)

**PDCA 四階段**：

```
┌─────────────────────────────────────┐
│  Plan (規劃)                         │
│  - 建立 ISMS 範圍                    │
│  - 風險評鑑 (識別、分析、評估)        │
│  - 選擇控制措施 (Annex A 114項)      │
│  - 制定資安政策與目標                │
└──────────┬──────────────────────────┘
           │
┌──────────▼──────────────────────────┐
│  Do (執行)                           │
│  - 實施控制措施                      │
│  - 資安教育訓練                      │
│  - 文件化程序                        │
│  - 資源分配                          │
└──────────┬──────────────────────────┘
           │
┌──────────▼──────────────────────────┐
│  Check (檢查)                        │
│  - 內部稽核                          │
│  - 管理審查                          │
│  - 監控與量測                        │
│  - 事件分析                          │
└──────────┬──────────────────────────┘
           │
┌──────────▼──────────────────────────┐
```

### 二、資安事件應變流程

**三階段應變**：

#### **事前 (準備)**
- 建立 CSIRT (資安事件應變小組)
- 制定應變計畫 (Incident Response Plan)
- 定期演練
- 建立通報機制

#### **事中 (應變)**
1. **偵測與通報**
2. **分類分級** (依影響程度)
3. **圍堵 (Containment)**
   - 隔離受感染系統
   - 阻斷惡意連線
4. **根除 (Eradication)**
   - 移除惡意程式
   - 修補漏洞
5. **復原 (Recovery)**
   - 系統還原
   - 驗證功能

#### **事後 (改善)**
- 事件分析報告
- 根因分析 (Root Cause Analysis)
- 更新政策與程序
- 教育訓練

**通報時限** (依資通安全管理法)：
- 一級事件：1 小時內
- 二級事件：8 小時內
- 三級事件：36 小時內

### 三、數位鑑識證據保全五步驟

```
1. 識別 (Identification)
   └─ 確認證據位置與類型
   
2. 蒐集 (Collection)
   └─ 使用適當工具取得證據
   
3. 擷取 (Acquisition)
   └─ 製作位元對位元複本 (Bit-by-bit copy)
   └─ 計算雜湊值 (MD5/SHA-256)
   
4. 封緘 (Preservation)
   └─ 證據袋封存
   └─ 填寫監管鏈表單 (Chain of Custody)
   
5. 運送 (Transportation)
   └─ 安全運送至鑑識實驗室
```

**證據同一性 (Identity)**：
確保法庭上的證據與現場取得的證據**完全相同**，未經竄改。

**證明方式**：
- **雜湊值比對** (原始 vs 複本)
- **監管鏈紀錄** (誰、何時、何地、做了什麼)

### 四、風險評鑑三階段

#### **1. 風險識別 (Risk Identification)**

**步驟**：
- 識別資產 (Asset)
- 識別威脅 (Threat)
- 識別弱點 (Vulnerability)

**範例**：
```
資產：客戶資料庫
威脅：SQL Injection 攻擊
弱點：未使用參數化查詢
```

#### **2. 風險分析與估計 (Risk Analysis)**

**風險值計算**：
```
風險值 = 威脅可能性 × 影響程度

範例：
威脅可能性: 高 (3)
影響程度: 極高 (5)
風險值 = 3 × 5 = 15 (高風險)
```

**常用矩陣**：

| 可能性 \ 影響 | 低 (1) | 中 (2) | 高 (3) | 極高 (4) |
|--------------|--------|--------|--------|-----------|
| **極高 (5)** | 5      | 10     | 15     | **20**    |
| **高 (4)**   | 4      | 8      | 12     | **16**    |
| **中 (3)**   | 3      | 6      | 9      | 12        |
| **低 (2)**   | 2      | 4      | 6      | 8         |

#### **3. 風險評估 (Risk Evaluation)**

**風險處理策略**：
1. **風險迴避** (Avoidance) - 停止該活動
2. **風險降低** (Mitigation) - 實施控制措施
3. **風險轉移** (Transfer) - 購買保險
4. **風險接受** (Acceptance) - 接受殘餘風險

#### ✍️ 標準答案示範

**解答**：

**一、ISO 27001 ISMS 與 PDCA 概述**

ISO/IEC 27001 是**國際資訊安全管理系統標準**，採用 **PDCA (Plan-Do-Check-Act)** 循環，系統化管理組織資訊安全。PDCA 是一種**持續改進**的管理方法，透過四階段循環不斷提升安全水準。

**PDCA 循環目標**：
- 建立系統化的管理流程
- 確保資安措施有效性
- 持續改進安全水準

---

**二、Plan (規劃階段) - 電子郵件安全政策制定**

**1. 建立 ISMS 範圍**

定義電子郵件安全管理的涵蓋範圍：
- **適用對象**：全體員工、外部合作夥伴
- **適用系統**：Exchange Server、Gmail for Business 等
- **保護資產**：Email 內容、附件、聯絡人清單

**2. 風險評鑑**

**資產識別**：
```
- 機密郵件 (合約、財務報表)
- 客戶資料庫匯出檔 (CSV 附件)
- 員工帳號密碼
```

**威脅與弱點識別**：

| 威脅 | 弱點 | 風險值 |
|------|------|--------|
| **釣魚郵件** | 員工安全意識不足 | 高 (15) |
| **商業郵件詐騙 (BEC)** | 未驗證主管郵件真偽 | 極高 (20) |
| **郵件竊聽** | 未加密傳輸 | 中 (9) |
| **附件惡意程式** | 未掃毒 | 高 (12) |

**3. 選擇控制措施**

依據風險評鑑結果，選擇 Annex A 控制措施：
- **A.9.4.1** 資訊存取限制
- **A.10.1.1** 密碼學控制政策
- **A.12.2.1** 惡意軟體防護
- **A.13.2.3** 電子通訊

**4. 制定電子郵件安全政策**

**政策內容**：
```
【電子郵件安全政策】

1. 禁止事項：
   - 禁止寄送機密資料至個人信箱
   - 禁止點擊可疑連結
   - 禁止開啟不明附件

2. 強制措施：
   - 外寄機密郵件須加密 (PGP/S/MIME)
   - 附件自動掃毒
   - 啟用 SPF/DKIM/DMARC 防範偽冒

3. 教育訓練：
   - 每季一次釣魚郵件演練
   - 新進員工資安課程
```

**目標設定**：
- 釣魚郵件識別率達 90%
- 郵件加密覆蓋率 100% (機密文件)
- 惡意附件攔截率 99.9%

**三、Do (執行階段) - 實施 Email 安全控制措施**

**1. 技術控制措施**

**A. Email Gateway (郵件閘道)**

部署企業級郵件安全閘道：
```
┌──────────────┐
│   網際網路    │
└──────┬───────┘
       │
┌──────▼───────┐
│ Email Gateway │
│ ┌──────────┐ │
│ │ Anti-Spam │ │ ← 垃圾郵件過濾
│ ├──────────┤ │
│ │ Anti-Virus│ │ ← 附件掃毒
│ ├──────────┤ │
│ │ Sandboxing│ │ ← 沙箱分析可疑附件
│ └──────────┘ │
└──────┬───────┘
       │
┌──────▼───────┐
│ Mail Server  │
│ (Exchange)   │
└──────────────┘
```

**B. 郵件加密**

實施 **S/MIME** 或 **PGP**：
```
寄件流程:
1. 使用者撰寫郵件
2. Outlook 自動偵測「機密」標籤
3. 使用收件者公鑰加密
4. 寄出加密郵件

收件流程:
1. 收件者下載加密郵件
2. 使用私鑰解密
3. 閱讀明文內容
```

**C. SPF/DKIM/DMARC 設定**

防範郵件偽冒：
```dns
; SPF 記錄 (允許哪些伺服器可寄信)
company.com. IN TXT "v=spf1 ip4:203.x.x.x include:_spf.google.com ~all"

; DKIM 記錄 (數位簽章)
default._domainkey.company.com. IN TXT "v=DKIM1; k=rsa; p=MIGfMA0GCS..."

; DMARC 政策 (如何處理驗證失敗郵件)
_dmarc.company.com. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc@company.com"
```

**五、Act (行動階段) - 持續改進**

**1. 矯正措施 (Corrective Action)**

**範例情境**：
```
稽核發現：
20% 員工點擊模擬釣魚郵件連結

根因分析：
- 訓練不足
- 郵件警示不明顯

矯正措施：
1. 增加訓練頻率：每季 → 每月
2. 部署郵件橫幅：
   [外部郵件] 此郵件來自組織外部，請謹慎處理
3. 啟用 ATP (Advanced Threat Protection)
```

**2. 預防措施 (Preventive Action)**

**趨勢分析**：
觀察到 **AI 深偽 (Deepfake)** 語音詐騙興起

**預防行動**：
- 建立「主管指示匯款」雙重驗證機制
- 更新政策：大額匯款需電話確認 + 簽核
- 員工教育：深偽技術識別

**3. 持續改進**

**每年檢討改進**：

```
2023 年度改進項目:

原政策：機密郵件手動加密
問題：遺漏加密風險
改進：自動偵測關鍵字自動加密

原狀況：員工點擊率 30%
訓練強化：增加實例教學
成果：點擊率降至 10%

技術升級：
- 舊系統：基礎防毒
- 新系統：AI 行為分析 (Sandbox)
```

**回到 Plan 階段**：
- 重新評估風險
- 更新控制措施清單
- 設定新年度目標

---

**六、結論**

透過 **PDCA 循環**應用於電子郵件安全管理：
- **Plan**：制定政策與風險評鑑
- **Do**：實施技術與管理控制
- **Check**：稽核與監控成效
- **Act**：持續改進與調整

PDCA 確保電子郵件安全**不是一次性專案**，而是**持續優化的過程**，隨著威脅演變不斷強化防護能力。這種系統化方法也可套用至其他資安領域 (如端點防護、網路安全等)。

### 題目 2：數位鑑識證據保全五步驟

#### 📖 原題 (106年高考二級)

> **題目**：為使執行資安事件調查時，能有效保全及運用數位證據，確保數位證據的同一性，請說明人員於執行數位證據識別、蒐集、擷取、封緘及運送，五個作業程序的內容為何？（25 分）

#### 🎯 答題架構分析

1.  **前言**：定義數位證據與同一性 (Chain of Custody) 的重要性。
2.  **五大作業程序詳解**：
    *   識別 (Identification)
    *   蒐集 (Collection)
    *   擷取 (Acquisition)
    *   封緘 (Preservation)
    *   運送 (Transportation)
3.  **結語**：強調程序正義與證據力。

#### 📊 評分建議 (預估配分 25 分)

**前言 (3 分)**
- 說明數位證據易被竄改的特性 (1 分)
- 定義「同一性」：法庭呈現證據 = 原始證據 (2 分)

**五大程序 (20 分，每項 4 分)**
- **識別**：確認範圍、現場封鎖、揮發性順序 (4 分)
- **蒐集**：錄影存證、斷網/斷電決策 (4 分)
- **擷取**：Bit-stream Copy、寫入阻斷器 (Write Blocker)、雜湊值計算 (4 分)
- **封緘**：防靜電袋、監管鏈 (Chain of Custody) 簽名 (4 分)
- **運送**：防震防磁、全程監控 (4 分)

**結語 (2 分)**
- 總結證據力關鍵在於程序的嚴謹性。

**答題提示**：
- ✅ 關鍵字必寫：**揮發性證據 (Volatile Evidence)**、**寫入阻斷器 (Write Blocker)**、**雜湊值 (Hash)**、**監管鏈 (Chain of Custody)**。
- ✅ 強調「擷取」階段必須製作複本，不可操作原始證物。

#### ✍️ 標準答案示範

**解答**：

**一、前言**

數位證據具有易揮發、易竄改之特性。為確保證據在法庭上具有證據能力 (Admissibility)，必須嚴格遵守標準作業程序 (SOP) 以確保證據的**同一性 (Identity)**，即證明法庭上提示的證據與犯罪現場原始取得的證據完全一致，未經任何竄改或汙染。

**二、數位證據保全五大作業程序**

**1. 識別 (Identification)**
此階段目標為確認證據的範圍與位置，並進行現場保全。
- **現場封鎖**：限制人員進出，避免破壞現場跡證。
- **證據識別**：確認數位裝置（電腦、手機、隨身碟）及雲端帳號。
- **揮發性順序**：依據證據揮發性高低決定優先順序（如：記憶體 > 暫存檔 > 硬碟 > 光碟）。
- **記錄現狀**：對螢幕畫面、線路連接狀況進行拍照或錄影存證。

**2. 蒐集 (Collection)**
使用適當工具與方法，將實體證物從現場移出或進行現場採證。
- **斷電決策**：
    - 若未加密：直接拔除電源（避免執行關機程序觸發惡意腳本）。
    - 若加密或為伺服器：考慮進行活體鑑識 (Live Forensics) 擷取記憶體。
- **實體蒐集**：將電腦主機、儲存媒體等證物進行標示與包裝。
- **錄影存證**：全程錄影蒐集過程，確保程序無瑕疵。

**3. 擷取 (Acquisition)**
將數位資料從儲存媒體中複製出來，重點在於「不更動原始證物」。
- **使用寫入阻斷器 (Write Blocker)**：確保在讀取過程中，絕對不會對原始硬碟寫入任何資料。
- **位元對位元複製 (Bit-stream Copy)**：製作與原始硬碟完全一樣的映像檔 (Image File)，包含已刪除但未被覆蓋的資料（Slack Space）。
- **計算雜湊值 (Hash Calculation)**：對原始證物與映像檔分別計算雜湊值 (如 MD5, SHA-256)，兩者必須一致，以證明證據同一性。

**4. 封緘 (Preservation)**
確保證物在保存過程中不受物理或環境破壞，並建立監管鏈。
- **防護包裝**：使用防靜電袋 (Anti-static bag)、防磁盒包裝硬碟，避免靜電或磁場破壞資料。
- **封條彌封**：在證物袋開口處貼上封條，並由鑑識人員簽名，確保未被私自開封。
- **監管鏈紀錄 (Chain of Custody)**：詳細記錄證物經手人、時間、地點、移交原因，確保證物保管連續性。

**5. 運送 (Transportation)**
將證物安全運送至鑑識實驗室。
- **環境控制**：運送過程需注意防震、防潮、防高溫及防強磁。
- **安全戒護**：專人專車運送，避免證物遺失或遭掉包。
- **入庫點交**：抵達實驗室後，立即進行點交並存入證物室（證物櫃）。

**三、結論**

數位鑑識的核心價值在於「程序正義」。唯有嚴格遵守上述五大程序，特別是透過**寫入阻斷**與**雜湊值比對**確保**同一性**，並透過**監管鏈**確保**連續性**，數位證據才能經得起法庭的檢驗，成為有效的定罪依據。

3.  **事件等級**：依機密性、完整性、可用性受損程度，區分為 1 至 4 級。
4.  **事件類型**：如分散式阻斷服務攻擊 (DDoS)、惡意程式、網頁置換、資料外洩等。
5.  **影響範圍與損害評估**：受影響之系統、資料筆數、預估損失金額或影響業務範圍。
6.  **已採取之應變措施**：目前已執行的緊急處置（如斷網、關機、阻擋 IP）。
7.  **外部支援需求**：是否需要上級機關或技術服務中心支援。

**二、公務機關資通安全事件通報及應變流程**

**1. 事件發現與等級研判**
- 機關自行發現或接獲外部情資（如 SOC 告警、民眾檢舉）。
- 初步研判事件影響程度，依規定判定為 1、2、3 或 4 級事件。

    - 上級機關。
    - 資通安全管理法主管機關（數位發展部資通安全署）。
    - 若涉及個資外洩，需同步通報個資主管機關。
    - 若涉及犯罪，應通報檢警單位。

**3. 應變作業 (Response)**
- **損害控制**：立即執行緊急應變措施（如隔離受駭主機、阻斷惡意連線），防止損害擴大。
- **完成時限**：
    - 1、2 級事件：**72 小時內**完成損害控制或復原。
    - 3、4 級事件：**36 小時內**完成損害控制或復原。
- **證據保全**：若涉及犯罪或需後續調查，應保存系統日誌 (Log) 及數位證據。

**4. 復原與結案 (Recovery & Closing)**
- **系統復原**：清除惡意程式、修補漏洞後，將系統回復正常運作。
- **提出報告**：
    - 1、2 級事件：於 **1 個月內**提交「調查、處理及改善報告」。
    - 3、4 級事件：於 **1 個月內**提交報告（主管機關可要求縮短期限）。
- **檢討改進**：針對事件根因進行檢討，修訂資安政策或加強防護措施，並納入年度演練項目。

### 題目 4：風險評鑑 (識別、估計、評估)

#### 📖 原題 (110年高考三級)

> **題目**：請回答下列資訊系統風險評鑑相關問題：
> (一) 請說明風險識別及其步驟。（8 分）
> (二) 請說明風險估計及其步驟。（8 分）
> (三) 請說明風險評估。（4 分）

#### 🎯 答題架構分析

1.  **風險管理流程**：識別 -> 分析(估計) -> 評估(評價)。
2.  **風險識別**：找出資產、威脅、弱點。
3.  **風險估計**：計算可能性 x 衝擊 = 風險值。
4.  **風險評估**：與風險胃納比較，決定處理優先順序。

#### 📊 評分建議 (預估配分 20 分)

**第一部分：風險識別 (8 分)**
- 定義：找出可能造成損失的要素。
- 步驟：資產盤點 -> 威脅識別 -> 弱點識別 -> 衝擊識別。

**第二部分：風險估計 (8 分)**
- 定義：量化或質化分析風險大小。
- 步驟：分析現有控制措施 -> 評估發生可能性 -> 評估衝擊程度 -> 計算風險值。

**第三部分：風險評估 (4 分)**
- 定義：將風險值與標準比較。
- 內容：決定風險等級 -> 決定處理優先順序 (接受/處理)。

**答題提示**：
- ✅ 記得公式：**風險 = 資產價值 × 威脅 × 弱點** (或 **可能性 × 衝擊**)。
- ✅ 關鍵詞：**資產 (Asset)**、**威脅 (Threat)**、**弱點 (Vulnerability)**。

#### ✍️ 標準答案示範

**解答**：

依據 ISO 27005 或 NIST SP 800-30 標準，風險評鑑 (Risk Assessment) 是資安管理的基礎，主要包含以下三個階段：

**一、風險識別 (Risk Identification)**

**1. 定義**
風險識別的目的是找出所有可能影響組織資訊安全目標（機密性、完整性、可用性）的潛在風險因素。

**2. 執行步驟**
1.  **資產盤點 (Asset Identification)**：識別組織內的重要資訊資產（如：客戶資料庫、伺服器、原始碼、專利文件）。
2.  **威脅識別 (Threat Identification)**：找出可能危害資產的潛在原因（如：駭客攻擊、惡意軟體、火災、人為疏失）。
3.  **弱點識別 (Vulnerability Identification)**：找出資產或控制措施的脆弱點（如：未修補的系統漏洞、密碼強度不足、機房無門禁）。
4.  **現有控制措施識別**：確認目前已實施的防護措施（如：已安裝防火牆、已定期備份）。

**二、風險估計 (Risk Estimation / Analysis)**

**1. 定義**
風險估計（或稱風險分析）是對已識別的風險進行定性或定量的分析，以決定風險的嚴重程度。

**2. 執行步驟**
1.  **評估發生可能性 (Likelihood)**：分析威脅利用弱點導致事件發生的機率（如：高、中、低）。
2.  **評估衝擊程度 (Impact)**：分析事件發生後對組織造成的損害程度（如：財務損失、商譽受損、營運中斷）。
3.  **計算風險值 (Risk Level)**：
    > **風險值 = 發生可能性 × 衝擊程度**
    
    例如：可能性為 3 (高)，衝擊為 3 (高)，則風險值為 9。

**三、風險評估 (Risk Evaluation)**

**1. 定義**
風險評估（或稱風險評價）是將計算出的風險值與組織的**風險準則 (Risk Criteria)** 或**風險胃納 (Risk Appetite)** 進行比較，以決定風險處理的優先順序。

**2. 執行內容**
1.  **風險分級**：將風險劃分為不同等級（如：不可接受風險、可容忍風險、可接受風險）。
2.  **決策依據**：
    - 若風險值 > 可接受水準：必須採取行動（如：降低、轉移、避免）。
    - 若風險值 < 可接受水準：可選擇風險接受 (Accept)，持續監控即可。
3.  **產出結果**：產出「風險評鑑報告」與「待處理風險清單」，作為後續風險處理計畫 (Risk Treatment Plan) 的輸入。

3. ❌ 缺少具體技術措施 (僅寫「加強訓練」太空泛)

#### 📘 ISO 27001 深度解析與實務指南

**什麼是 ISO 27001？為什麼重要？**

ISO 27001 是由**國際標準組織 (ISO)** 與**國際電工委員會 (IEC)** 共同制定的**資訊安全管理系統 (ISMS)** 國際標準。它不是一套技術規範，而是一套**管理框架**，幫助組織系統化地保護資訊資產。

**重要性**：
1. **國際認可**：全球超過 60,000 家組織通過認證
2. **客戶信任**：證明組織有能力保護客戶資料
3. **法規遵循**：滿足 GDPR、台灣個資法等法律要求
4. **降低風險**：系統化識別並管理資安風險
5. **商業優勢**：政府標案、企業合作常要求 ISO 27001 認證

---

**ISO 27001 vs 其他標準**

| 標準 | 目的 | 差異 |
|------|------|------|
| **ISO 27001** | 資訊安全**管理系統**認證標準 | 可驗證、可認證（須第三方稽核） |
| **ISO 27002** | 資訊安全**控制措施**實務守則 | 指引性質，不可認證 |
| **NIST Cybersecurity Framework** | 美國網路安全框架 | 以風險管理為核心，5大功能 |
| **CIS Controls** | 關鍵安全控制措施 | 技術導向，20項具體控制 |

**簡單記法**：
- ISO 27001 = **管理框架** (怎麼管)
- ISO 27002 = **技術指引** (怎麼做)

---

**PDCA 循環詳解：為何是持續改進的關鍵？**

PDCA 又稱為**戴明環 (Deming Cycle)**，由管理學大師 W. Edwards Deming 提出，核心概念是**持續改進 (Continuous Improvement)**。

**為何資安需要 PDCA？**

傳統「一次性專案」的問題：
```
傳統做法:
建置防火牆 → 完成 → 結案 ❌

問題:
- 新威脅出現 → 防護措施過時
- 員工離職 → 政策無人遵守
- 設備老化 → 無人維護
```

**PDCA 解決方案**：
```
PDCA 循環:
規劃防護 → 執行措施 → 檢查成效 → 改進不足 → 
  ↑                                            ↓
  └────────────── 持續循環 ──────────────────────┘
```

**每個階段的核心問題**：

**Plan (規劃)**：我們要保護什麼？風險在哪裡？
- 核心產出：風險評鑑報告、控制措施清單
- 常見錯誤：範圍定義過大（想保護一切）或過小（遺漏關鍵系統）

**Do (執行)**：我們如何實施保護措施？
- 核心產出：已部署的技術控制、已執行的訓練
- 常見錯誤：買了工具但沒有正確設定（如防火牆規則過於寬鬆）

**Check (檢查)**：保護措施有效嗎？
- 核心產出：稽核報告、監控數據
- 常見錯誤：稽核流於形式（只查文件，不驗證實際狀態）

**Act (行動)**：我們如何持續改進？
- 核心產出：矯正措施、預防措施
- 常見錯誤：發現問題但沒有資源改善

---

**Annex A 控制措施深入理解**

ISO 27001:2013 Annex A 包含 **14 個領域、114 項控制措施**。這是**必須評估**的控制清單，但不是全部都要實施（依風險決定）。

**14 大領域概覽**：

```
A.5  資訊安全政策 (2項)
     └─ 管理階層承諾、政策審查

A.6  資訊安全組織 (7項)
     └─ 組織架構、職責分工、行動裝置政策

A.7  人力資源安全 (6項)
     └─ 背景查核、離職程序、保密協議

A.8  資產管理 (10項)
     └─ 資產清冊、資料分類、媒體處置

A.9  存取控制 (14項) ⭐ 最多控制項
     └─ 使用者管理、密碼政策、權限管理

A.10 密碼學 (2項)
     └─ 加密政策、金鑰管理

A.11 實體與環境安全 (15項)
     └─ 門禁管制、監視器、設備保護

A.12 作業安全 (14項)
     └─ 變更管理、備份、日誌管理、惡意軟體防護

A.13 通訊安全 (7項)
     └─ 網路隔離、資料傳輸加密、電子郵件安全

A.14 系統取得、開發與維護 (13項)
     └─ 安全開發生命週期、測試、變更控制

A.15 供應商關係 (5項)
     └─ 供應商評估、服務水準協議、外包管理

A.16 資訊安全事件管理 (7項)
     └─ 通報機制、應變程序、證據保全

A.17 營運持續管理 (4項)
     └─ BCP/DR、備援、演練

A.18 遵循性 (8項)
     └─ 法規識別、個資保護、稽核
```

**如何選擇控制措施？**

**範例情境**：中小企業電商平台

```
風險評鑑發現：
威脅：SQL Injection 攻擊
資產：客戶資料庫 (100萬筆個資)
影響：洩密 → 罰款 + 商譽損失

選擇控制措施：
✓ A.9.4.1 - 存取限制 (限制資料庫管理員人數)
✓ A.12.6.1 - 技術弱點管理 (定期漏洞掃描)
✓ A.14.2.1 - 安全開發政策 (使用 Prepared Statements)
✓ A.16.1.1 - 事件通報 (建立資料外洩應變計畫)

不選擇：
✗ A.11.1.2 - 實體門禁 (資料庫在雲端，無實體機房)
✗ A.15.1.1 - 供應商政策 (此風險與供應商無關)
```

**關鍵原則**：
- 不是 114 項都要做！
- 根據風險評鑑結果，選擇適用的控制措施
- 未選擇的控制要有合理理由（記錄在「適用性聲明 SOA」）

**實務案例分析**

**案例 1：某醫院導入 ISO 27001 失敗經驗**

**背景**：
- 某區域醫院為符合「醫療機構電子病歷製作及管理辦法」，決定導入 ISO 27001
- 預算 300 萬，時程 1 年

**失敗原因**：
1. **高層支持不足**：
   - 院長認為「只是拿張證書」，未參與管理審查
   - 預算僅給顧問費，沒有資源改善設備

2. **範圍定義錯誤**：
   - 將「全院所有系統」納入範圍 → 太大難以管理
   - 應聚焦「電子病歷系統」即可

3. **文件過度複雜**：
   - 政策文件 200 頁，護理人員根本不會看
   - 應簡化為「一頁政策 + 分級SOP」

4. **稽核流於形式**：
   - 內部稽核只查文件，不查實際運作
   - 外稽前 2 週「惡補文件」，通過後又回復原狀

**結果**：
- 花 300 萬取得證書
- 但實際資安能力未提升
- 1 年後發生勒索病毒事件 → 證明 ISMS 形同虛設

**教訓**：
- ✅ ISO 27001 不只是「拿證書」，而是「建立文化」
- ✅ 高層必須真正參與，不能只掛名
- ✅ 範圍適中，品質優於規模
- ✅ 稽核要「查實際，不只查文件」

---

**案例 2：某金融科技公司成功導入**

**背景**：
- 新創金融科技公司，客戶要求 ISO 27001 認證才簽約
- 預算有限，僅 50 萬

**成功關鍵**：

1. **範圍精準**：
   - 僅納入「支付 API 服務」
   - 排除內部 CRM、財務系統

2. **善用雲端優勢**：
   - 使用 AWS → 直接繼承 AWS 的 ISO 27001 證書（共同責任模型）
   - 僅需管理「應用層」安全

3. **自動化為主**：
   - 使用 GitHub Actions 自動化漏洞掃描
   - 使用 AWS CloudWatch 自動化日誌監控
   - 省下人力成本

4. **文件簡化**：
   - 政策文件控制在 30 頁內
   - 使用 Markdown 版控，方便更新

**成果**：
- 6 個月取得認證
- 成本控制在 50 萬內
- 客戶簽約金額 500 萬 → ROI 10倍

**教訓**：
- ✅ 範圍越小越好（尤其新創）
- ✅ 善用雲端服務商的認證（不要重複造輪子）
- ✅ 自動化 > 人力
- ✅ ISO 27001 是商業工具，不是成本中心

**常見誤解與澄清**

| 誤解 | 真相 |
|------|------|
| 「通過 ISO 27001 就不會被駭」 | ❌ ISO 27001 是管理制度，不保證絕對安全。它降低風險，但無法消除風險 |
| 「114 項控制措施都要做」 | ❌ 根據風險評鑑選擇適用的控制，合理排除不適用項目 |
| 「只有大企業才需要」 | ❌ 新創、中小企業也可導入，範圍可小可大，依需求調整 |
| 「通過認證就完成了」 | ❌ 認證有效期 3 年，每年需監督稽核，且需持續維護 ISMS |
| 「外部認證比內部稽核重要」 | ❌ 內部稽核才是日常運作的關鍵，外稽只是驗證 |
| 「買了資安設備就符合標準」 | ❌ 技術只是一部分，還需政策、訓練、監控、改進 |

**常見面試/考試題目解析**

**Q1: ISO 27001 與 ISO 27002 的差異？**

**答**：
- **ISO 27001**：**要求標準**（What），定義 ISMS 必須包含哪些要素，可認證
- **ISO 27002**：**實務守則**（How），提供控制措施的實施指引，不可認證
- **關係**：27001 的 Annex A 列出控制措施，27002 說明如何實施這些控制

**Q2: PDCA 中，Check 和 Act 有何不同？**

**答**：
- **Check（檢查）**：**發現問題**，透過稽核、監控發現現狀與目標的落差
- **Act（行動）**：**改進問題**，針對 Check 發現的問題採取矯正/預防措施
- **關鍵差異**：Check 是「診斷」，Act 是「治療」

**Q3: 為何 ISO 27001 強調「持續改進」而非「一次達標」？**

**答**：
因為威脅環境持續變化：
- 新漏洞持續被發現
- 攻擊手法不斷演進
- 組織業務持續變動
- 法規要求持續更新

**一次性達標**只能應對當下風險，**持續改進**才能應對未來挑戰。

#### ✍️ 標準答案示範

**解答**：

**一、數位鑑識與證據同一性概述**

**數位鑑識 (Digital Forensics) 定義**：
運用**科學方法**，從數位設備中**識別、保全、分析、呈現**數位證據，用於**法律訴訟或事件調查**，確保證據具有法律效力。

**證據同一性 (Identity of Evidence)**：
確保法庭上呈現的證據與**現場取得的證據完全相同**，未經任何竄改、遺失或污染。這是證據具有**證據能力 (Admissibility)** 的關鍵要素。

**證明方式**：
1. **雜湊值比對**：證明位元完全一致
2. **監管鏈紀錄**：證明保管過程無缺失
3. **時間戳記**：證明證據取得時間

**為何重要**：
- 刑事案件：駭客入侵、詐騙、竊取商業機密
- 民事案件：勞資糾紛、智財侵權
- 內部調查：員工洩密、違規使用

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
編號  證物類型           位置                  狀態
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
E001  嫌疑人電腦         辦公桌 (開機中)       ⚠ 須即時保全記憶體
E002  外接硬碟 (USB)     抽屜                  ✓ 關機狀態
E003  手機 (iPhone)      嫌疑人口袋            ⚠ 須飛航模式
E004  公司伺服器         機房                  ✓ 需製作映像檔

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**B. 證據類型分類**

**揮發性證據 (Volatile Evidence)**：
- **RAM 記憶體**：關機即消失 → **最優先保全**
- **網路連線狀態**：進程、開啟檔案
- **暫存檔案**

**非揮發性證據 (Non-Volatile Evidence)**：
- 硬碟、SSD
- USB 隨身碟
- 雲端儲存 (Dropbox, Google Drive)

**保全優先順序** (依揮發性排序)：
```
優先度 1 (最易消失):
  └─ CPU 快取、記憶體

優先度 2:
  └─ 暫存檔、Swap 分區

優先度 3:
  └─ 硬碟、外接儲存

優先度 4:
  └─ 備份、日誌檔
```

**C. 現場記錄**

**拍照/錄影**：
- 電腦螢幕畫面 (避免關機後遺失)
- 設備連接狀態 (網路線、USB 插孔)
- 房間整體環境

**範例**：
```
時間：2024-11-28 14:30
地點：3F 行銷部 座位 3-A15
發現：嫌疑人電腦螢幕顯示 WinRAR 壓縮客戶資料
狀態：電腦開機中，滑鼠游標停在桌面
```

### **步驟 2：蒐集 (Collection)**

**目的**：
使用**正確方法**取得證據，避免破壞原始狀態。

**作業內容**：

**A. 活體取證 (Live Forensics)**

**情境**：電腦處於**開機狀態**

**為何不能直接關機**：
- 記憶體中的惡意程式會消失
- 加密磁碟可能無法再存取 (BitLocker/FileVault)
- 暫存資料遺失

**記憶體擷取工具**：

| 工具 | 適用系統 | 功能 |
|------|---------|------|
| **FTK Imager** | Windows | 擷取 RAM、製作映像檔 |
| **Volatility** | Windows/Linux/macOS | 記憶體分析框架 |
| **WinPmem** | Windows | 開源記憶體擷取工具 |

**步驟**：
```bash
# 1. 插入乾淨 USB (寫入阻擋器)
# 2. 執行 FTK Imager (免安裝版)
# 3. Capture Memory → 選擇輸出路徑
# 4. 記錄以下資訊：
   - RAM 大小: 16 GB
   - 擷取時間: 2024-11-28 14:35:22
   - MD5: a3f5e8c9...
```

**B. 死體取證 (Dead Forensics)**

**情境**：電腦已關機或需關機

**正確關機程序**：
1. **拍照螢幕** (若仍開機)
2. **拔除電源** (不透過作業系統關機，避免觸發反鑑識腳本)
3. **標記所有連接線**

**不當作法** ❌：
- 直接點「關機」→ 可能觸發資料自毀程式
- 隨意拔線 → 無法還原連接狀態

**C. 網路證據蒐集**

**防火牆/IDS 日誌**：
```bash
# 匯出特定時間區間
iptables -L -n -v > firewall_log_$(date +%Y%m%d).txt

# SIEM 查詢 (Splunk 範例)
index=security sourcetype=firewall 
earliest="2024-11-28:08:00:00" 
latest="2024-11-28:18:00:00"
| table _time, src_ip, dst_ip, action
```

**封包擷取** (若事件發生中)：
```bash
# tcpdump 擷取封包
sudo tcpdump -i eth0 -w evidence_$(date +%Y%m%d_%H%M%S).pcap
```

### **步驟 3：擷取 (Acquisition)**

**目的**：
製作**位元對位元 (Bit-by-bit)** 的完整複本，確保原始證據不被變動。

**作業內容**：

**A. 硬碟映像檔製作**

**寫入阻擋器 (Write Blocker)**：
```
原理：硬體裝置，僅允許「讀取」，禁止「寫入」

┌──────────┐      ┌──────────────┐      ┌──────────┐
│ 證據硬碟  │ ──→ │ Write Blocker │ ──→ │ 鑑識電腦  │
│ (唯讀)   │      │   (硬體保護)   │      │          │
└──────────┘      └──────────────┘      └──────────┘
```

**為何需要**：
即使掛載硬碟**只是讀取**，作業系統也可能自動寫入 (如 Windows 自動建立 Thumbs.db)，破壞證據原始狀態。

**製作工具**：

| 工具 | 類型 | 優點 |
|------|------|------|
| **dd (Linux)** | 命令列 | 免費、標準格式 |
| **FTK Imager** | GUI | 易用、支援多種格式 |
| **EnCase** | 商業軟體 | 法庭認可度高 |

**dd 指令範例**：
```bash
# 製作映像檔
sudo dd if=/dev/sdb of=/mnt/evidence/disk_image.dd bs=4M status=progress

# 計算雜湊值
md5sum disk_image.dd > disk_image.dd.md5
sha256sum disk_image.dd > disk_image.dd.sha256
```

**B. 雜湊值計算**

**目的**：
產生證據的**數位指紋**，任何微小變動都會導致雜湊值改變。

**常用演算法**：
- **MD5**：128-bit (易碰撞，僅輔助用)
- **SHA-1**：160-bit (已棄用)
- ⭐ **SHA-256**：256-bit (目前主流)
- **SHA-512**：512-bit (最安全)

**範例**：
```
原始證據硬碟:
SHA-256: 3f4a8b9c2e1d6f8a7c5b4d3e2f1a9b8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2a1

複製的映像檔:
SHA-256: 3f4a8b9c2e1d6f8a7c5b4d3e2f1a9b8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2a1
         ↑ 完全一致 → 證明複本與原件相同
```

**C. 文件記錄**

建立**擷取紀錄表**：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
證據編號: E001
證據描述: 嫌疑人電腦硬碟 (Seagate 1TB)
序號: ST1000DM003-1CH162
擷取時間: 2024-11-28 15:10 ~ 16:25
擷取人員: 李大華 (鑑識專員)
工具: FTK Imager 4.7.1
映像檔: E001_disk.E01
原始雜湊 (SHA-256): 3f4a8b9c...
映像雜湊 (SHA-256): 3f4a8b9c...
驗證結果: ✓ 一致
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### **步驟 4：封緘 (Preservation)**

**目的**：
確保證據在運送與保管過程中**不被竄改或遺失**。

**作業內容**：

**A. 證據袋封存**

**標準證據袋**：
```
┌─────────────────────────────────────┐
│  證據編號: E001                      │
│  案件編號: 2024-CS-1128              │
│  ┌──────────────────────────────┐   │
│  │  證物: 硬碟 (1TB Seagate)     │   │
│  │  取得日期: 2024-11-28        │   │
│  │  取得地點: 3F 行銷部          │   │
│  └──────────────────────────────┘   │
│                                     │
│  封緘簽名: ____________ 日期:____   │
│  (封條貼此處，撕開即失效)            │
└─────────────────────────────────────┘
```

**封緘要求**：
- **一次性封條**：撕開後無法復原
- **簽名見證**：取證人員 + 在場證人
- **防水防震**：使用適當包材

**B. 監管鏈表單 (Chain of Custody)**

**定義**：
記錄證據從**取得到法庭呈現**的完整移轉過程。

**監管鏈範例**：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
證據編號: E001 | 證物: 嫌疑人電腦硬碟
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
序號 | 日期時間           | 移交人   | 接收人   | 目的         | 簽名
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 1   | 2024-11-28 16:30  | 李大華   | 王小明   | 運送至實驗室 | ✓
 2   | 2024-11-28 17:00  | 王小明   | 實驗室   | 開始分析     | ✓
 3   | 2024-11-29 09:00  | 實驗室   | 證物室   | 分析完成封存 | ✓
 4   | 2024-12-05 14:00  | 證物室   | 法庭     | 出庭作證     | ✓
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**為何重要**：
缺少任何一環，辯護律師可質疑：「證據在某段時間無人看管，可能遭竄改！」→ 證據失去效力

**C. 儲存環境**

**實體證物儲存**：
- **溫控環境**：避免硬碟物理損壞
- **防潮箱**：防止受潮
- **門禁管制**：僅授權人員可進入證物室

**數位檔案儲存**：
- **多份備份**：至少 3 份 (原件 + 2 備份)
- **離線儲存**：避免網路攻擊
- **加密保護**：AES-256 加密

---

### **步驟 5：運送 (Transportation)**

**目的**：
將證據**安全運送**至鑑識實驗室或法庭，避免運送過程損壞。

**作業內容**：

**A. 運送前檢查**

**檢查清單**：
- ✓ 證據已封緘
- ✓ 監管鏈表單填寫完整
- ✓ 雜湊值已記錄
- ✓ 包裝防震防水

**B. 運送方式**

**短距離運送** (同城市)：
- 專人護送
- 使用公務車
- 避免交由快遞 (監管鏈中斷)

**長距離運送**：
- 保全公司運送
- 全程 GPS 追蹤
- 保險

**C. 交接程序**

**到達實驗室**：
```
1. 核對證據編號
2. 檢查封條完整性
3. 拍照記錄
4. 雙方簽收監管鏈表單
5. 重新計算雜湊值 → 確認未變動
```

**不當作法** ❌：
- 快遞寄送 → 監管鏈中斷
- 未簽收紀錄 → 交接不明
- 未驗證雜湊值 → 無法證明未竄改

---

**三、結論**

數位證據保全五步驟環環相扣：
1. **識別**：找出所有證據
2. **蒐集**：正確取得證據
3. **擷取**：製作完整複本 + 雜湊值
4. **封緘**：封存 + 監管鏈
5. **運送**：安全交付目的地

任何一步疏失，都可能導致證據在法庭上**被質疑真實性**，進而失去證據能力。

**關鍵原則**：
- **保持原始狀態** (使用寫入阻擋器)
- **可複製驗證** (雜湊值)
- **完整記錄** (監管鏈)

**實務案例**：

**案例 1：台灣首例勒索病毒刑案 (2017)**
- 嫌犯電腦中發現比特幣錢包私鑰
- 鑑識人員製作完整映像檔，並計算 SHA-256
- 法庭上，辯護律師質疑證據真實性
- 檢方出示監管鏈紀錄 + 雜湊值比對 → 證據獲採納

**案例 2：某公司內部洩密案**
- 員工 USB 隨身碟包含客戶資料
- 但因鑑識人員**未使用寫入阻擋器**，直接插入電腦
- Windows 自動在 USB 建立 `System Volume Information` 資料夾
- 辯護律師主張：「USB 已被寫入，證據遭污染！」
- 法官認定證據**不具證據能力**

---

**常見錯誤**：

| 錯誤行為 | 後果 | 正確做法 |
|---------|------|---------|
| 直接開嫌疑人電腦查看 | 存取時間被修改 → 證據污染 | 先擷取記憶體，再製作映像檔 |
| 未計算雜湊值 | 無法證明未竄改 | 每次移轉都重新驗證 |
| 監管鏈有空白 | 證據失效 | 完整記錄每次移交 |
| 使用個人 USB 擷取證據 | 可能帶入病毒 | 使用乾淨、專用儲存設備 |

- 通報機制（1 分）
  - 內部通報、主管機關通報

**事後改進（3 分）**
- 事後檢討會議（1 分）
- 更新應變計畫（1 分）
- 備份與 BCP 整合（1 分）

**答題提示**：
- ✅ 必須舉**實際案例**（WannaCry、台塑等）
- ✅ 影響要分**技術、法律、商譽、財務**四面向
- ✅ 改善建議要具體且可執行
- ⚠️ 這是管理類重點題，要全面且實務


#### ✍️ 標準答案示範

**解答**：

**一、缺乏資安事件應變計畫的影響**

資安事件應變計畫 (Incident Response Plan, IRP) 是組織**事先規劃**如何應對資安事件的完整流程。若無有效應變計畫，將造成以下嚴重影響：

**1. 技術層面影響**

| 影響 | 說明 | 範例 |
|------|------|------|
| **應變時間延遲** | 不知從何開始處理，錯失黃金搶救時間 | 勒索病毒蔓延全公司，因不知該先斷網還是先備份 |
| **損失擴大** | 未及時圍堵，導致災情擴散 | 一台主機中毒 → 2天後全公司 500 台電腦加密 |
| **證據遺失** | 未保全證據，無法追查攻擊來源 | 員工重灌電腦 → 惡意程式消失 → 無法分析 |
| **復原困難** | 沒有備份或備份也被加密 | 備份與正式環境在同一網段 → 同時被加密 |

**2. 法律合規影響**

**台灣法規要求**：
```
資通安全管理法第 18 條:
「公務機關應訂定資通安全事件通報及應變機制」

個人資料保護法第 12 條:
「資料外洩應於知悉後 72 小時內通報」
```

**違法後果**：
- 未通報 → 罰款 10-100 萬
- 資料外洩未採取適當措施 → 最高罰 1,500 萬（個資法第 47 條）
- 客戶集體訴訟 → 賠償金額更高

**3. 商譽損失**

**信任危機**：
```
事件發生 → 媒體報導 → 客戶恐慌 → 大量退訂

範例：某電商平台資料外洩
- 媒體報導：100 萬筆客戶資料外洩
- 客戶反應：3天內退訂率飆升 40%
- 股價：一週內下跌 25%
- 商譽修復：至少 2 年
```

**長期影響**：
- 新客戶不願加入
- 合作夥伴終止契約
- 政府標案被拒

**4. 財務衝擊**

**直接成本**：

| 項目 | 金額估算 | 說明 |
|------|---------|------|
| 勒索贖金 | 數百萬～數千萬 | Bitcoin 支付，不保證解密 |
| 緊急應變成本 | 50-200 萬 | 請資安公司處理 |
| 系統重建 | 100-500 萬 | 重灌、資料復原 |
| 法律費用 | 50-300 萬 | 訴訟、和解 |

**間接成本**：
- 業務中斷損失：每天數百萬～數千萬
- 客戶賠償：依合約條款
- 客戶流失：難以估計

**範例：某中小企業遭勒索病毒**
```
直接成本:
- 贖金: 300 萬 (50 BTC)
- 資安顧問: 80 萬
- 系統重建: 150 萬
- 小計: 530 萬

間接成本:
- 業務中斷 7 天: 700 萬
- 客戶流失 (20%): 1,200 萬/年
- 商譽損失: 難以估計

總損失: 超過 2,400 萬
```

**二、實際案例分析**

**案例 1：台灣某醫院勒索病毒事件 (2020)**

**事件經過**：
```
Day 1 (週五晚上 8 點):
└─ IT 人員發現多台電腦檔案被加密

Day 1 (晚上 9 點):
└─ 院長被通知，但不知如何處理
└─ 決定「週一再找資安公司」

Day 2-3 (週末):
└─ 病毒持續蔓延
└─ 醫護人員仍使用電腦，幫助病毒傳播

Day 4 (週一早上):
└─ 全院 90% 電腦無法開機
└─ 掛號系統、醫療紀錄全部無法存取
└─ 醫院宣布「暫停門診」
```

**影響**：
- **醫療服務中斷** 5 天
- **3,000 位病患** 無法就診
- **損失金額** 約 500 萬（醫療收入 + 復原成本）
- **信譽受損** 媒體大幅報導

**失敗原因**：
1. ❌ 無應變計畫 → 不知該聯絡誰（資安公司、警察、衛福部？）
2. ❌ 無隔離機制 → 病毒週末持續蔓延
3. ❌ 備份失效 → 備份伺服器也被加密
4. ❌ 無演練經驗 → 院長、IT、醫護各行其是

---

**案例 2：國際案例 - Colonial Pipeline 勒索軟體攻擊 (2021, 美國)**

**事件規模**：
- 美國最大燃油管線營運商遭 DarkSide 勒索軟體攻擊
- 供應美國東岸 45% 燃油
- 關閉管線 6 天
- 支付贖金 440 萬美元 (75 BTC)

**為何損失慘重**：
1. **無 OT/IT 網路隔離**：辦公室網路 → 工業控制系統
2. **無應變演練**：管理層不知是否該關閉管線
3. **決策緩慢**：花 2 天才決定支付贖金

**教訓**：
- 即使大企業，若無有效應變計畫，一樣會癱瘓
- 工業控制系統 (OT) 必須與 IT 網路隔離
- 事前演練至關重要

**三、改善建議：建立完整資安事件應變計畫**

### **階段 1：事前準備 (Preparation)**

**1.1 建立 CSIRT (資安事件應變小組)**

**組織架構**：
```
┌─────────────────────────────────────┐
│        CSIRT 指揮官 (CISO)           │
│        - 統籌指揮                    │
│        - 對外發言                    │
└──────────┬──────────────────────────┘
           │
    ┌──────┴──────┬──────────┬─────────┐
    │             │          │         │
┌───▼────┐  ┌────▼───┐  ┌───▼────┐  ┌──▼───┐
│技術組  │  │法務組  │  │公關組  │  │業務組│
│- 圍堵  │  │- 通報  │  │- 聲明  │  │- BCP │
│- 復原  │  │- 合規  │  │- 客服  │  │      │
└────────┘  └────────┘  └────────┘  └──────┘
```

**職責分工**：

| 角色 | 職責 | 聯絡方式 |
|------|------|---------|
| **CSIRT 指揮官** | 統籌指揮、決策、對外發言 | 24hr 電話: 0912-XXX-XXX |
| **技術組長** | 技術分析、圍堵、復原 | IT 經理 |
| **法務組長** | 法規通報、訴訟處理 | 法務長 |
| **公關組長** | 媒體聲明、客戶溝通 | 行銷長 |
| **業務組長** | 啟動 BCP、客戶安撫 | 營運長 |

**1.2 制定應變程序文件**

**文件清單**：
```
✓ 事件分類分級標準
✓ 緊急聯絡人清單
✓ 技術處理 SOP (按事件類型)
✓ 對外聲明範本
✓ 法規通報流程
✓ 供應商合約 (資安公司、公關公司)
```

**事件分級範例**：

| 等級 | 定義 | 範例 | 應變時間 |
|------|------|------|---------|
| **Level 1 (重大)** | 影響核心業務、資料外洩 | 勒索病毒、100萬筆個資外洩 | 1小時內啟動 |
| **Level 2 (高)** | 影響部分業務 | 網站被駭、部門資料庫入侵 | 4小時內啟動 |
| **Level 3 (中)** | 影響單一系統 | 單台主機中毒 | 8小時內處理 |
| **Level 4 (低)** | 無實質影響 | 釣魚郵件但未點擊 | 記錄備查 |

**1.3 定期演練**

**演練類型**：

**桌上演練 (Tabletop Exercise)**：
```
頻率: 每季一次
時間: 2-3 小時
方式: 模擬情境討論

情境範例:
「週五下午 5 點，會計收到疑似 CEO 郵件要求匯款，
 並附帶惡意連結，已有 3 人點擊...」

討論:
- 誰負責通知 IT？
- 該不該立即斷網？
- 如何確認有多少人點擊？
- 週末如何聯絡 CSIRT 成員？
```

**實兵演練 (Full-Scale Exercise)**：
```
頻率: 每年一次
時間: 半天～1天
方式: 

實際模擬攻擊

範例:
1. 資安公司配合，發動真實攻擊
2. CSIRT 在不知情下應變
3. 事後檢討缺失
```

**1.4 準備技術工具**

**必備工具清單**：
```
✓ 備份系統 (3-2-1 原則: 3份/2種媒體/1份離線)
✓ SIEM (日誌集中管理)
✓ EDR (端點偵測與回應)
✓ 隔離環境 (Sandbox)
✓ 鑑識工具 (FTK Imager, Volatility)
✓ 離線 USB (內含應急工具)
```

### **階段 2：事中應變 (Response)**

**事件應變六步驟**：

```
1. 偵測 → 2. 通報 → 3. 分類 → 4. 圍堵 → 5. 根除 → 6. 復原
```

**步驟 1：偵測 (Detection)**

**偵測來源**：
- SIEM 告警
- 防毒軟體通知
- 員工通報異常
- 客戶投訴

**步驟 2：通報 (Notification)**

**內部通報**：
```
發現異常 (IT 人員)
   ↓
通知 CSIRT 指揮官 (15分鐘內)
   ↓
召集 CSIRT 小組
   ↓
啟動應變程序
```

**外部通報** (依法規)：
- **資安法**：特定公務機關須通報主管機關
- **個資法**：資料外洩須通報當事人與主管機關
- **金融業**：須通報金管會

**步驟 3：分類分級 (Classification)**

**快速評估**：
```
1. 影響範圍？(單一系統 vs 全公司)
2. 資料類型？(機密 vs 公開)
3. 業務影響？(核心 vs 邊緣)

→ 決定事件等級 (Level 1-4)
```

**步驟 4：圍堵 (Containment)**

**短期圍堵**：
```
目標: 立即阻止擴散

勒索病毒範例:
✓ 隔離受感染主機 (拔網路線)
✓ 停用共享資料夾
✓ 封鎖 C&C 伺服器 IP
✓ 停用被盜帳號
```

**長期圍堵**：
```
目標: 確保不再發生

✓ 部署 EDR
✓ 加強監控
✓ 修補漏洞
```

**步驟 5：根除 (Eradication)**

**清除威脅**：
```
✓ 移除惡意程式
✓ 修補被利用的漏洞
✓ 重置所有密碼
✓ 重建受損系統
```

**步驟 6：復原 (Recovery)**

**系統復原**：
```
1. 從乾淨備份還原
2. 驗證系統功能
3. 加強監控 (防止復發)
4. 分階段上線
```

### **階段 3：事後檢討 (Post-Incident)**

**3.1 事件報告**

**報告內容**：
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
資安事件調查報告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

一、事件摘要
- 發生時間: 2024-11-28 14:30
- 發現時間: 2024-11-28 15:00 (延遲 30 min)
- 事件類型: 勒索病毒 (Ryuk)
- 影響範圍: 50 台電腦

二、時間軸
14:30 - 員工點擊釣魚郵件
14:35 - 病毒開始加密
15:00 - IT 發現異常
15:15 - 啟動應變程序
15:30 - 隔離受感染區域
17:00 - 威脅清除

三、根因分析 (Root Cause)
- 直接原因: 員工點擊釣魚郵件
- 根本原因: 
  1. 郵件閘道未攔截
  2. 端點防護未更新
  3. 員工訓練不足

四、改善措施
- 短期 (1 週內):
  ✓ 更新郵件閘道規則

  ✓ 部署 EDR

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**檢討會議**：

```
時間: 事件結束後 1 週內
參加: CSIRT 全員

討論重點:
1. ✅ 做得好的地方 (繼續保持)
2. ❌ 做得不好的地方 (需改進)
3. 📝 應變流程修正
```

- 修訂應變程序
- 更新緊急聯絡人
- 補充新型攻擊手法
- 新增處理案例

**四、結論**

缺乏有效的資安事件應變計畫，將導致：
1. **技術影響**：損失擴大、證據遺失、復原困難
2. **法律風險**：未依法通報、巨額罰款
3. **商譽損失**：客戶流失、股價下跌
4. **財務衝擊**：直接成本 + 間接成本，動輒千萬

**建立完整應變計畫的關鍵**：
- ✅ **事前準備**：建立 CSIRT、制定程序、定期演練
- ✅ **事中應變**：快速偵測、有效圍堵、完整復原
- ✅ **事後檢討**：根因分析、經驗學習、持續改進

**黃金原則**：
> 「平時多流汗，戰時少流血」
> 定期演練 > 事後補救

**解答**：

**前言：風險評鑑概述**

**風險評鑑 (Risk Assessment)** 是 ISO 27001 ISMS 中 **PDCA 的 Plan 階段**核心工作，目的是**系統化地識別、分析、評估**資訊安全風險，作為選擇控制措施的依據。

**風險評鑑三階段**：

```
┌─────────────────────────────────────────┐
│  階段 1：風險識別 (Risk Identification)  │
│  └─ 找出有哪些風險                        │
└──────────┬──────────────────────────────┘
           │
┌──────────▼──────────────────────────────┐
│  階段 2：風險分析 (Risk Analysis)        │
│  └─ 評估風險的可能性與影響程度             │
└──────────┬──────────────────────────────┘
           │
┌──────────▼──────────────────────────────┐
│  階段 3：風險評估 (Risk Evaluation)      │
│  └─ 決定風險是否可接受、需要處理          │
└──────────────────────────────────────────┘
```

**國際標準依據**：
- **ISO 27005**：Information Security Risk Management
- **NIST SP 800-30**：Guide for Conducting Risk Assessments

---

**一、風險識別 (Risk Identification)**

**定義**：
系統化地找出組織面臨的**資訊安全威脅**，並識別可能被利用的**弱點**。

**目標**：
建立**完整的風險清單**，避免遺漏重大風險。

**步驟 1：識別資產 (Asset Identification)**

**1.1 資產清單建立**

**資產分類**：

| 資產類別 | 範例 | 評估重點 |
|---------|------|---------|
| **資料資產** | 客戶資料庫、財務報表、商業機密 | 機密性、完整性 |
| **軟體資產** | ERP系統、CRM系統、作業系統 | 可用性、完整性 |
| **硬體資產** | 伺服器、網路設備、儲存設備 | 可用性 |
| **服務資產** | 雲端服務、ISP服務、資安服務 | 可用性、機密性 |
| **人員資產** | IT人員、管理階層、外包商 | 知識、權限 |
| **實體資產** | 機房、辦公室、紙本文件 | 實體安全 |

**資產價值評估**：

**方法 1：定性評估**
```
資產價值等級：
- 極高 (Critical): 損失影響營運存續
- 高 (High): 嚴重影響業務
- 中 (Medium): 部分業務受影響  
- 低 (Low): 輕微影響
```

**方法 2：定量評估**
```
資產價值 = 重置成本 + 營運損失 + 商譽損失

範例：客戶資料庫
- 重置成本: 500萬 (重建資料庫)
- 營運損失: 2,000萬/年 (客戶流失)
- 商譽損失: 5,000萬 (品牌受損)
- 總價值: 7,500萬
```

**實務範例**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
資產清單表
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
資產編號 | 資產名稱        | 類別 | 價值等級 | 負責人
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
A001     | 客戶資料庫      | 資料 | 極高     | DBA
A002     | ERP系統         | 軟體 | 高       | IT經理
A003     | 核心網路交換器  | 硬體 | 高       | 網管
A004     | AWS雲端服務     | 服務 | 中       | 雲端管理員
A005     | 機房            | 實體 | 高       | 設施經理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**步驟 2：識別威脅 (Threat Identification)**

**2.1 威脅來源分類**

**人為威脅**：
- **惡意**：駭客、內部員工、競爭對手
- **非惡意**：人為疏失、誤操作

**環境威脅**：
- 火災、水災、地震
- 停電、溫度異常

**技術威脅**：
- 硬體故障
- 軟體漏洞
- 網路中斷

**2.2 常見威脅清單**

| 威脅類型 | 具體威脅 | 攻擊目標 |
|---------|---------|---------|
| **網路攻擊** | DDoS、APT、勒索病毒 | 系統可用性、資料機密性 |
| **資料外洩** | SQL Injection、內部洩密 | 資料機密性 |
| **實體破壞** | 火災、竊盜、破壞 | Hardware、機房 |
| **社交工程** | 釣魚郵件、假冒主管 | 帳號密碼、敏感資料 |
| **供應鏈** | 第三方廠商被駭 | 所有連接系統 |

**步驟 3：識別弱點 (Vulnerability Identification)**

**3.1 弱點來源**

**技術弱點**：
```
- 軟體漏洞 (未更新patch)
- 預設密碼未更改
- 加密強度不足
- 存取控制缺失
```

**管理弱點**：
```
- 缺乏資安政策
- 員工訓練不足
- 職責分工不清
- 備份機制缺失
```

**實體弱點**：
```
- 門禁管制不足
- 無監視器
- 機房溫控異常
- 電力備援缺失
```

**3.2 弱點識別方法**

| 方法 | 工具/技術 | 適用範圍 |
|------|----------|---------|
| **漏洞掃描** | Nessus, OpenVAS | 網路、系統漏洞 |
| **滲透測試** | Metasploit, Kali Linux | 實際攻擊模擬 |
| **組態稽核** | CIS Benchmarks | 系統組態檢查 |
| **人員訪談** | 問卷、訪談 | 管理面弱點 |
| **實地查核** | 現場檢查 | 實體弱點 |

**實務範例**：

```

風險識別表 (以客戶資料庫為例)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
資產: A001 客戶資料庫 (價值: 極高)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
威脅          | 弱點              | 風險描述
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**二、風險分析 (Risk Analysis / Risk Estimation)**

**定義**：
評估已識別風險的**發生可能性 (Likelihood)** 與**影響程度 (Impact)**，計算**風險值 (Risk Level)**。

**目標**：
量化或定性地評估風險大小，作為優先處理的依據。

**步驟 1：評估威脅發生可能性 (Likelihood Assessment)**

**1.1 定性評估**

| 等級 | 描述 | 發生頻率 | 範例 |
|------|------|---------|------|
| **極高 (5)** | 幾乎確定發生 | 每月 | 釣魚郵件攻擊 |
| **高 (4)** | 很可能發生 | 每季 | 軟體漏洞被利用 |
| **中 (3)** | 可能發生 | 每年 | 內部員工洩密 |
| **低 (2)** | 不太可能 | 每3年 | 重大火災 |
| **極低 (1)** | 罕見 | 5年以上 | 隕石撞擊 |

**1.2 可能性評估因子**

```
可能性 = f(威脅能力, 威脅動機, 弱點嚴重性, 現有控制措施)

範例：SQL Injection 風險
- 威脅能力: 高 (駭客工具易取得)
- 威脅動機: 高 (客戶資料可變現)
- 弱點嚴重性: 高 (未參數化查詢)
- 現有控制: 低 (僅基礎防火牆)
→ 綜合評估: 可能性 = 高 (4)
```

---

**步驟 2：評估影響程度 (Impact Assessment)**

**2.1 CIA 三要素影響**

**機密性 (Confidentiality) 影響**：
```
極高: 大量客戶個資外洩
高:   部分機密文件外洩
中:   內部文件外洩
低:   公開資訊外洩
極低: 無機密資訊
```

**完整性 (Integrity) 影響**：
```
極高: 財務資料遭竄改
高:   重要系統組態被修改
中:   一般文件被修改
低:   非關鍵資料錯誤
```

**可用性 (Availability) 影響**：
```
極高: 核心系統停機 > 24小時
高:   核心系統停機 8-24小時
中:   部分系統停機
低:   單一使用者無法存取
```

**2.2 業務影響評估 (BIA - Business Impact Analysis)**

| 影響面向 | 極高 (5) | 高 (4) | 中 (3) | 低 (2) | 極低 (1) |
|---------|---------|--------|--------|--------|---------|
| **財務損失** | >1,000萬 | 500-1,000萬 | 100-500萬 | 10-100萬 | <10萬 |
| **法律責任** | 刑事責任 | 鉅額罰款 | 行政罰鍰 | 警告 | 無 |
| **商譽損失** | 媒體頭條 | 客戶大量流失 | 部分客戶不滿 | 內部影響 | 無影響 |
| **營運中斷** | 停業 | >7天 | 3-7天 | 1-3天 | <1天 |

**實務範例**：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
影響程度評估 (客戶資料外洩情境)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
機密性影響: 極高 (100萬筆個資外洩)
財務損失:   極高 (罰款1,500萬 + 訴訟2,000萬)
商譽損失:   極高 (媒體大幅報導)
法律責任:   高 (個資法重罰)

綜合評估: 影響程度 = 極高 (5)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

**步驟 3：計算風險值 (Risk Calculation)**

**3.1 風險值公式**

```
風險值 (Risk Level) = 可能性 (Likelihood) × 影響程度 (Impact)

範例：SQL Injection 風險
- 可能性: 高 (4)
- 影響程度: 極高 (5)
- 風險值: 4 × 5 = 20 (極高風險)
```

**3.2 風險矩陣 (Risk Matrix)**

```
             影響程度 (Impact)

        ───┼─────┼─────┼─────┼─────┼─────┤
可    5    │  5  │ 10  │ 15  │ 20  │ 25  │
能 度 4    │  4  │  8  │ 12  │ 16  │ 20  │
性    3    │  3  │  6  │  9  │ 12  │ 15  │
      2    │  2  │  4  │  6  │  8  │ 10  │
      1    │  1  │  2 │  3  │  4  │  5  │
        
風險等級分類：
🔴 極高風險 (20-25): 立即處理
🟠 高風險 (12-16):   優先處理
🟡 中風險 (6-10):    計劃處理
🟢 低風險 (1-5):     可接受或監控
```

**三、風險評估 (Risk Evaluation)**

**定義**：
將**風險分析結果**與組織的**風險接受準則 (Risk Acceptance Criteria)** 比較，決定風險是否可接受、需要處理。

**目標**：
- 識別需要處理的風險
- 確定風險處理優先順序

**步驟 1：設定風險接受準則**

**範例 1：簡單準則**
```
風險值 ≥ 12  → 不可接受，必須處理
風險值 6-11  → 可接受，但需監控
風險值 ≤ 5   → 可接受
```

**範例 2：分層準則**

| 風險值 | 決策 | 必要行動 |
|-------|------|---------|
| **20-25** | 不可接受 | 立即停止相關業務，緊急處理 |
| **12-16** | 高階主管決策 | 提報董事會，制定處理計畫 |
| **6-10** | 部門主管決策 | 實施控制措施，定期監控 |
| **1-5** | 可接受 | 記錄備查，年度檢討 |

---

**步驟 2：風險評估決策**

**評估流程**：

```
┌──────────────────────┐
│  風險分析結果         │
│  (風險值: 20)        │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  比對風險接受準則     │
│  (風險值 ≥ 12 → 不可接受) │
└──────────┬───────────┘
           │
```


---

**步驟 3：風險處理策略選擇**

**四種風險處理策略**：

**1. 風險降低 (Risk Mitigation)**
```
定義: 實施控制措施，降低可能性或影響

範例：SQL Injection (風險值: 20)
措施:
- 使用參數化查詢 (降低可能性)
- 部署WAF (降低可能性)
- 資料庫加密 (降低影響)
- 定期滲透測試 (監控)

殘餘風險: 6 (可接受)
```

**2. 風險迴避 (Risk Avoidance)**
```
定義: 停止會產生風險的活動

範例：第三方支付API風險 (風險值: 18)
措施:
- 停用第三方支付
- 改用自建支付系統

殘餘風險: 0 (風險消除)
```

**3. 風險轉移 (Risk Transfer)**
```
定義: 將風險轉移給第三方承擔

範例：資料中心火災風險 (風險值: 10)
措施:
- 購買財產保險
- 使用雲端服務 (AWS/Azure)

殘餘風險: 3 (財務風險轉移給保險公司)
```

**4. 風險接受 (Risk Acceptance)**
```
定義: 接受風險，不採取額外措施

範例：舊版系統漏洞 (風險值: 5)
措施:
- 系統即將下線 (3個月後)
- 成本效益考量，不修補

殘餘風險: 5 (接受)
```

**策略選擇決策樹**：

```
                風險可接受？
                 │
        ┌────No──┴──Yes─┐
        │                │
    實施處理策略      接受風險
        │                └─→ [記錄 & 監控]
        ├─ 成本 < 損失？
        │   ├─ Yes → 降低
        │   └─ No  → 轉移
        │
        ├─ 可停止活動？
        │   └─ Yes → 迴避
        │
        └─ 有保險/外包？
            └─ Yes → 轉移
```

---

**四、結論**

**風險評鑑三階段總結**：

1. **風險識別**：
   - 識別資產、威脅、弱點
   - 建立完整風險清單
   - 輸出：風險識別表

2. **風險分析（估計）**：
   - 評估可能性與影響程度
   - 計算風險值
   - 輸出：風險矩陣、風險優先順序

3. **風險評估**：
   - 比對風險接受準則
   - 決定是否處理
   - 選擇風險處理策略
   - 輸出：風險處理計畫

**關鍵成功因素**：
- ✅ 高階主管支持
- ✅ 跨部門協作（IT、法務、業務、財務）
- ✅ 定期更新（至少每年一次）
- ✅ 文件化記錄

**風險評鑑循環**：

```
風險評鑑 → 風險處理 → 監控 → 再評鑑 → ...
   ↑                                  │
   └──────────── 持續改進 ─────────────┘
```

**實務案例分析**

**案例：某金融科技公司風險評鑑實務**

**背景**：
- 提供線上借貸服務
- 客戶資料 50萬筆
- 受金管會監管

**風險評鑑執行**：

**階段 1：風險識別**
```
資產識別:
- A001: 客戶個資資料庫 (極高)
- A002: 信用評分系統 (高)
- A003: 金流串接API (高)
- A004: 官網 (中)

威脅識別:
- T001: SQL Injection
- T002: DDoS攻擊
- T003: 內部洩密
- T004: 第三方API漏洞

弱點識別:
- V001: 未使用WAF
- V002: 員工權限過大
- V003: 第三方安全性未驗證
```

**階段 2：風險分析**
```
風險 R001: SQL Injection 攻擊客戶資料庫
- 可能性: 高 (4) - 外部測試發現弱點
- 影響: 極高 (5) - 50萬筆個資外洩
- 風險值: 20 (極高風險)

風險 R002: DDoS 攻擊官網
- 可能性: 中 (3) - 業界常見攻擊
- 影響: 高 (4) - 業務中斷
- 風險值: 12 (高風險)

風險 R003: 內部員工洩密
- 可能性: 中 (3) - 權限管理不佳
- 影響: 極高 (5) - 大量資料外洩
- 風險值: 15 (高風險)
```

**階段 3：風險評估與處理**
```
風險接受準則: 風險值 ≥ 12 → 不可接受

處理策略:
R001 (20): 降低
  → 實施參數化查詢
  → 部署WAF (Cloudflare)
  → 資料庫加密
  → 殘餘風險: 6

R002 (12): 轉移
  → 購買Cloudflare DDoS防護服務
  → 殘餘風險: 4

R003 (15): 降低
  → 實施最小權限原則
  → 部署DLP (Data Loss Prevention)
  → 每季權限審查
  → 殘餘風險: 6
```

**結果**：
- 投資成本：280萬（WAF + DLP + Cloudflare）
- 風險降低：3個極高/高風險 → 可接受範圍
- 通過金管會資安檢查

**常見錯誤與注意事項**

**常見錯誤**：

| 錯誤 | 後果 | 正確做法 |
|------|------|---------|
| 僅評估技術風險，忽略管理風險 | 員工洩密等風險未被識別 | 全面評估（技術/管理/實體） |
| 風險評鑑一次性，未定期更新 | 新威脅未納入 | 至少每年一次，重大變更時加做 |
| 高層未參與 | 風險接受準則不合理 | 高階主管必須參與決策 |
| 僅依賴工具掃描 | 管理面弱點遺漏 | 結合工具 + 訪談 + 實地查核 |
| 風險值計算主觀 | 不同人評估結果差異大 | 建立明確評估標準、多人交叉驗證 |
