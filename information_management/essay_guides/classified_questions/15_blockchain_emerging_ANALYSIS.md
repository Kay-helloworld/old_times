## 📚 核心知識解析

### 一、區塊鏈 (Blockchain) 完整解析

#### (1) 區塊鏈定義

**區塊鏈 (Blockchain)**: 一種**分散式帳本技術 (Distributed Ledger Technology, DLT)**,將交易資料打包成「區塊」(Block),透過**密碼學技術**將區塊按時間順序串連成「鏈」(Chain),資料儲存於**多個節點**,具有**去中心化、不可竄改、透明**等特性。

**核心概念**:
- 📦 **區塊 (Block)**: 資料容器,含多筆交易
- ⛓️ **鏈 (Chain)**: 透過Hash串連
- 🌐 **分散式 (Distributed)**: 不依賴單一中心
- 🔒 **密碼學 (Cryptography)**: Hash+數位簽章

---

#### (2) 區塊結構 (Block Structure)

每個區塊包含以下資料:

```
┌──────────────────────────────────┐
│        區塊 #N (Block #N)         │
├──────────────────────────────────┤
│ 區塊標頭 (Block Header)           │
│  ├─ 版本號 (Version)              │
│  ├─ 前一區塊Hash (Previous Hash) ◄─┐
│  ├─ 默克爾樹根 (Merkle Root)      │ │
│  ├─ 時間戳 (Timestamp)            │ │
│  ├─ 難度目標 (Difficulty)         │ │
│  └─ Nonce (隨機數)                │ │
├──────────────────────────────────┤ │
│ 交易資料 (Transactions)           │ │
│  ├─ 交易1: A → B (1 BTC)          │ │
│  ├─ 交易2: C → D (0.5 BTC)        │ │
│  └─ 交易3: ...                    │ │
└──────────────────────────────────┘ │
            Hash (區塊N)             │
                 ↓                   │
┌──────────────────────────────────┐ │
│        區塊 #N+1                  │ │
│  前一區塊Hash = Hash(區塊N) ─────┘
│  ...                              │
└──────────────────────────────────┘
```

**關鍵欄位說明**:

| 欄位 | 說明 | 作用 |
|------|------|------|
| **前一區塊Hash** | 前一個區塊的Hash值 | 串連成鏈,確保順序 |
| **Merkle Root** | 所有交易的摘要 | 快速驗證交易完整性 |
| **Timestamp** | 區塊產生時間 | 時間戳,確保順序 |
| **Nonce** | 隨機數 | PoW挖礦時不斷調整,尋找符合難度的Hash |

---

#### (3) 區塊鏈如何確保「不可竄改」?

**核心機制**: **Hash函數 + 前一區塊Hash串連**

**情境**: 駭客想竄改歷史交易

```
【原始區塊鏈】
區塊100: "A付給B 10元"
         Hash100 = 0x1234
            ↓
區塊101: 前Hash = 0x1234, "C付給D 5元"
         Hash101 = 0x5678
            ↓
區塊102: 前Hash = 0x5678, ...

【駭客嘗試竄改區塊100】
修改: "A付給B 10元" → "A付給B 1000元"
   ↓
Hash100改變: 0x1234 → 0xABCD (不同!)
   ↓
區塊101的「前Hash = 0x1234」不符!
→ 驗證失敗,整條鏈從區塊101開始斷裂

【駭客必須重新計算】
1. 重算區塊100 (找新Nonce)
2. 重算區塊101 (前Hash改為0xABCD)
3. 重算區塊102...
4. 重算所有後續區塊 (可能數千萬個!)

【問題】
其他節點都有正確的區塊鏈
→ 駭客的鏈會被拒絕 (不是最長鏈)
```

**結論**: 要竄改歷史必須:
1. 重新計算該區塊及**所有後續區塊**
2. 算得比全網路**還快** (掌握51%算力)
3. 成本極高,**幾乎不可能**

#### (4) 共識機制 (Consensus Mechanism)

**問題**: 分散式系統如何就「哪個區塊有效」達成共識?

**方案1: PoW (Proof of Work, 工作量證明)**

**原理**: 透過**計算難題**競爭記帳權

```
【挖礦流程】
1. 礦工蒐集待處理交易
2. 組成新區塊
3. 不斷調整Nonce,計算Hash
4. 目標: 找到Hash < 難度目標

範例:
Hash(區塊) = SHA256(區塊資料 + Nonce)
目標: Hash必須以10個0開頭
      0x0000000000......  (極難!)

第一個找到的人 → 獲得記帳權 + 獎勵(BTC)
其他節點驗證 → 接受新區塊
```

**優點**:
- ✅ 安全性高 (攻擊成本=重算所有區塊)
- ✅ 完全公開,任何人可參與

**缺點**:
- ❌ 耗電 (比特幣年耗電 > 阿根廷全國)
- ❌ 速度慢 (Bitcoin: 7 TPS, Ethereum 1.0: 15 TPS)
- ❌ 51%攻擊風險 (掌握過半算力可竄改)

**代表**: Bitcoin, Ethereum 1.0

---

**方案2: PoS (Proof of Stake, 權益證明)**

**原理**: 透過**持有代幣數量**選出驗證者

```
【驗證流程】
1. 驗證者質押代幣 (Stake)
2. 系統隨機選出驗證者 (質押越多,機率越高)
3. 驗證者驗證交易,產生新區塊
4. 其他驗證者確認
5. 驗證者獲得交易手續費

【懲罰機制 (Slashing)】
如果驗證者作惡 (如雙重簽名)
→ 質押的代幣被沒收 (Slash)
```

**優點**:
- ✅ 節能 (不需挖礦)
- ✅ 速度快 (Ethereum 2.0: 1000-10000 TPS)
- ✅ 攻擊成本高 (需持有大量代幣)

**缺點**:
- ❌ 「富者越富」(大戶更易被選中)
- ❌ 技術複雜

**代表**: Ethereum 2.0 (The Merge, 2022年轉PoS), Cardano

#### (5) 區塊鏈三種類型

**類型1: 公有鏈 (Public Blockchain)**

**定義**: **完全開放**,任何人可讀取、交易、參與共識

**特性**:
- 完全去中心化
- 高度透明
- 匿名性

**代表**:
- **Bitcoin**: 第一個區塊鏈,數位黃金
- **Ethereum**: 智慧合約平台

**適用**: 加密貨幣、NFT

---

**類型2: 私有鏈 (Private Blockchain)**

**定義**: **權限控制**,僅特定組織可存取

**特性**:
- 中心化程度高
- 速度快 (不需複雜共識)
- 隱私性高

**代表**:
- **Hyperledger Fabric**: IBM主導,企業級區塊鏈

**適用**: 企業內部,需保密

---

**類型3: 聯盟鏈 (Consortium Blockchain)**

**定義**: **半去中心化**,由多個組織共同管理

**特性**:
- 部分去中心化
- 速度快
- 可控的透明度

**代表**:
- **Quorum**: JP Morgan開發,金融聯盟
- **R3 Corda**: 銀行聯盟

**適用**: 金融聯盟、供應鏈管理

---

**三種鏈比較**:

| 面向 | 公有鏈 | 聯盟鏈 | 私有鏈 |
|------|--------|--------|--------|
| **參與者** | 任何人 | 經授權組織 | 單一組織 |
| **去中心化** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| **速度** | 慢 (7-15 TPS) | 中 (100s TPS) | 快 (1000s TPS) |
| **隱私** | 低 (公開透明) | 中 (可設定) | 高 (完全控制) |
| **信任模型** | 無需信任 | 部分信任 | 完全信任 |
| **應用** | 加密貨幣、NFT | 銀行聯盟、供應鏈 | 企業內部稽核 |

**特性1: 去中心化 (Decentralization)**

```

所有人 → 中央伺服器 (銀行)
```

### 二、NFT (Non-Fungible Token, 非同質化代幣)

#### (1) NFT 定義

**NFT**: 建構在區塊鏈上的**獨一無二**數位資產,每個NFT都有唯一識別碼,**不可互換** (Non-Fungible)。

**同質化 vs 非同質化**:

| 特性 | 同質化 (Fungible) | 非同質化 (Non-Fungible) |
|------|------------------|------------------------|
| **定義** | 可互換, 無差異 | 獨一無二,不可互換 |
| **範例** | 鈔票(100元就是100元) | 藝術品(每幅畫不同) |
| **區塊鏈** | 加密貨幣 (BTC, ETH) | **NFT** |
| **可分割** | 可 (0.5 BTC) | 不可 (你無法持有0.5個NFT) |

**NFT vs 加密貨幣**:

```
【加密貨幣 (如Bitcoin)】
├─ 同質化: 你的1 BTC = 我的1 BTC
├─ 可分割: 可轉0.001 BTC
└─ 用途: 支付、儲值

【NFT】
├─ 非同質化: NFT #1234 ≠ NFT #5678
├─ 不可分割: 只能整個轉移
└─ 用途: 數位收藏品、所有權證明
```

---

#### (2) NFT、區塊鏈、加密貨幣的關係 (111年關務考題!)

```
┌─────────────────────────────────────┐
│       區塊鏈 (Blockchain)            │
│       底層技術平台                   │
└────────────┬────────────────────────┘
             │
    ┌────────┴────────┐
    ↓                 ↓
┌─────────┐      ┌─────────┐
│ 加密貨幣 │      │   NFT   │
│(Crypto) │      │(Non-Fun)│
└─────────┘      └─────────┘
    ↓                 ↓
【支付媒介】      【所有權憑證】
同質化代幣        非同質化代幣
可分割            不可分割
```

**關係說明**:

1. **區塊鏈** = 底層技術 (如Ethereum平台)
2. **加密貨幣** = 同質化代幣 (如ETH, USDT)
3. **NFT** = 非同質化代幣 (建構在區塊鏈上)

**比喻**:
```
區塊鏈 = 手機作業系統 (iOS)
```

### 三、智慧合約 (Smart Contract)

#### (1) 智慧合約定義 (106年地特考題!)

**智慧合約 (Smart Contract)**: 部署在區塊鏈上的**自動執行程式**,當預設條件達成時**自動執行**,無需人工介入。

**核心特性**:
- 🤖 **自動執行**: 條件符合→自動執行
- 🔒 **不可竄改**: 部署後無法修改
- 🌐 **去中心化**: 在區塊鏈上運行
- 💰 **可處理資產**: 可接收/發送加密貨幣

---

#### (2) 智慧合約 vs 一般應用程式

| 面向 | 一般應用程式 | 智慧合約 |
|------|------------|---------|
| **執行環境** | 中央伺服器 (如AWS) | **區塊鏈** (分散式) |
| **可修改性** | 可隨時更新 | **部署後不可變** |
| **信任需求** | 需要信任開發者/公司 | **無需信任** (Code is Law) |
| **執行成本** | 伺服器費用 | **Gas Fee** (每次執行都要付) |
| **執行速度** | 快 | 慢 (需等區塊確認) |
| **透明度** | 程式碼可能不公開 | **程式碼公開** (可驗證) |
| **停機風險** | 伺服器當機→無法使用 | **7x24運作** (分散式) |

**範例**:

```
【一般App: 自動扣款】
銀行App設定: 每月1號扣1000元
問題: 需信任銀行 (可能誤扣)

【智慧合約: 自動扣款】
合約部署在Ethereum:
IF (今天是1號) AND (帳戶餘額 >= 1000)
THEN (自動轉帳1000 DAI給房東)

特點:
✅ 公開透明 (任何人可驗證程式碼)
✅ 自動執行 (無需信任)
❌ 一旦部署,無法修改 (要小心Bug!)
```

---

#### (3) 智慧合約應用

**應用1: DeFi (去中心化金融)**

```
【借貸協議 (如Aave)】
1. 你抵押 10 ETH
2. 智慧合約自動放貸 → 你獲得 5000 USDT
3. 你還款 + 利息 → 合約自動歸還ETH

無需銀行,完全自動化!
```

**應用2: 去中心化交易所 (DEX)**

```
【Uniswap】
智慧合約管理流動性池
├─ 自動計算匯率
├─ 自動執行交易
└─ 手續費自動分配給流動性提供者
```

---

**應用3: DAO (去中心化自治組織)**

```
【投票治理】
智慧合約定義規則:
IF (提案獲得>50%投票) THEN (自動執行提案)

範例:
社群投票是否將資金投資於項目X
→ 過半同意 → 智慧合約自動轉帳
```

### 四、零信任架構 (Zero Trust Architecture) - 114年新增熱門考題!

#### (1) 零信任定義與核心理念

**零信任 (Zero Trust)**: 一種安全理念,假設**網路內外都不可信**,**永不信任,始終驗證** (Never Trust, Always Verify)。

**核心原則**:

```
傳統「城堡與護城河」模型:
    外部(不信任) | 內部(信任)
                防火牆
    ❌ 問題: 內部威脅、橫向移動

零信任模型:
    所有人/設備 → 都要驗證
    即使在「內網」→ 也不自動信任
```

**與傳統邊界防禦的差異**:

| 面向 | 傳統邊界防禦 | 零信任架構 |
|------|------------|-----------|
| **信任模型** | 內網=可信 | **永不信任** |
| **驗證時機** | 入網時驗證一次 | **持續驗證** |
| **存取控制** | 網段級 | **資源級** (微分段) |
| **預設權限** | 一旦進入,可存取大部分資源 | **最小權限** |
| **適用情境** | 固定辦公室 | **遠端辦公、雲端** |

---

#### (2) 零信任三大關鍵技術 (112/114年高考必考!)

**技術1: 使用者身分鑑別 (User Identification)**

**至少3種方法**:

| 方法 | 說明 | 範例 |
|------|------|------|
| **多因素驗證 (MFA)** | 密碼 + OTP/生物辨識 | Google Authenticator |
| **單一簽入 (SSO)** | 一次登入,存取多系統 | Microsoft Entra ID (Azure AD) |
| **無密碼驗證** | 生物辨識/FIDO2 | Windows Hello, Face ID |
| **憑證式驗證** | PKI數位憑證 | 自然人憑證 |
| **行為分析** | AI分析使用者行為 | 異常登入時間/地點→警報 |

**詳細說明**:

```
【MFA (Multi-Factor Authentication)】
三種因素:
├─ Something you know (知道): 密碼
├─ Something you have (持有): 手機OTP、USB Key
└─ Something you are (生物): 指紋、臉部

零信任要求: 至少2個因素
```

**技術2: 設備鑑別 (Device Authentication)**

**至少2種方法**:

| 方法 | 說明 | 優點 |
|------|------|------|
| **設備憑證** | 每個設備有唯一數位憑證 | 無法偽造 |
| **設備指紋 (Device Fingerprint)** | 收集設備特徵 (OS版本、硬體) | 識別可疑設備 |
| **健康檢查 (NAC)** | 檢查防毒、Patch是否最新 | 確保設備安全 |
| **信任平台模組 (TPM)** | 硬體安全晶片 | 硬體級安全 |
| **MDM整合** | Mobile Device Management | 管控行動裝置 |

**詳細說明**:

```
【NAC (Network Access Control)】
設備要連網前,先檢查:
├─ 防毒軟體是否啟用?
├─ OS Patch是否最新?
├─ 有無越獄/Root?
└─ 符合標準 → 允許連線
    不符合 → 隔離/僅存取修補區
```

**技術3: 信任推斷 (Trust Inference)**

**定義**: **動態評估**使用者/設備的風險等級,決定存取權限

**評估因素**:

```
風險評分計算:
├─ 登入時間: 半夜2點登入 → +10分(可疑)
├─ 登入地點: 從未去過的國家 → +20分
├─ 設備: 新設備 → +15分
├─ 行為: 大量下載檔案 → +25分
└─ 總分 > 50 → 要求重新驗證/MFA

風險等級 → 存取權限:
低風險 (0-30): 正常存取
中風險 (30-60): 限制敏感資料
高風險 (60+): 僅唯讀/要求MFA
```

**實例**:

```
【情境】
員工A平常在台北辦公室
今天突然從美國登入
    ↓
信任推斷評估:
├─ 地點異常 (+30分)
├─ 設備新 (+20分)
├─ 風險分數: 50
    ↓
動作: 要求額外MFA驗證
     + 通知資安團隊
```

---

#### (3) 零信任實施架構

**核心組件**:

```
┌──────────────────────────────────┐
│   使用者 / 設備                  │
└────────────┬─────────────────────┘
             ↓
┌──────────────────────────────────┐
│    身分驗證 (Identity Provider)  │
│    - MFA                         │
│    - SSO                         │
└────────────┬─────────────────────┘
             ↓
┌──────────────────────────────────┐
│   策略引擎 (Policy Engine)        │
│   - 評估風險                     │
│   - 決定權限                     │
└────────────┬─────────────────────┘
             ↓
┌──────────────────────────────────┐
│   微分段 (Micro-Segmentation)    │
│   - 資源隔離                     │
└────────────┬─────────────────────┘
             ↓
┌──────────────────────────────────┐
│      受保護資源                  │
│      (應用程式、資料)            │
└──────────────────────────────────┘
```

**關鍵實踐**:

**1. 微分段 (Micro-Segmentation)**

```
【傳統】
內網一大片,資料庫伺服器↔檔案伺服器可互通

【微分段】
每個應用/資料庫都隔離
Web伺服器 → (嚴格規則) → DB伺服器
            僅允許必要連線

效益: 駭客即使入侵Web,也無法橫向移動到DB
```

**2. 最小權限原則 (Least Privilege)**

```
員工只能存取「工作必需」的資源
範例:
├─ HR人員: 只能存取HR系統
├─ 工程師: 只能存取開發環境
└─ 即使都在「內網」,也互不相通
```

**3. 持續驗證 (Continuous Verification)**

```
不是「登入一次就信任一整天」
而是「每隔一段時間重新驗證」

範例:
├─ 每30分鐘: 重新評估風險
├─ 行為異常: 立即要求重新驗證
└─ Session自動過期
```

### 五、後量子密碼 (Post-Quantum Cryptography, PQC) - 114年超熱門!

#### (1) 量子電腦威脅

**量子電腦原理**:

```
【傳統電腦】
位元 (Bit): 0 或 1

【量子電腦】
量子位元 (Qubit): 可同時是 0 和 1 (疊加態)
n個Qubit → 2^n種狀態同時計算

範例:
3個傳統位元 → 一次計算1種狀態 (如 101)
3個量子位元 → 一次計算2³=8種狀態
```

**Shor演算法 (1994年提出)**:

```
【問題】
RSA加密基於「大數分解困難」
範例: 將 15 = 3 × 5 容易
       但 將 N (300位數) = p × q 極難

【Shor演算法】
量子電腦可在多項式時間內分解大數!
→ RSA、ECC都將被破解
```

**威脅時程**:

```
專家預估:
├─ 2030年: 量子電腦可能破解RSA-2048
├─ 2040年: 量子電腦商用化
└─ 「收割後解密」攻擊:
    駭客今天蒐集加密資料
    → 等10年後量子電腦成熟
    → 解密今天的資料
```

---

#### (2) 後量子密碼 (PQC) 定義

**PQC**: 能抵抗**量子電腦攻擊**的密碼演算法,基於**數學難題**(即使量子電腦也難解)。

**NIST PQC標準化 (2024年公布)**:

| 類別 | 演算法 | 用途 |
|------|--------|------|
| **加密/密鑰交換** | CRYSTALS-Kyber | 替代RSA/ECC |
| **數位簽章** | CRYSTALS-Dilithium | 替代RSA/ECDSA簽章 |
| **數位簽章** | FALCON | 替代RSA/ECDSA簽章 |
| **數位簽章** | SPHINCS+ | 備用方案 |

**PQC演算法類型**:

1. **格密碼 (Lattice-based)**: CRYSTALS-Kyber, Dilithium
2. **雜湊密碼 (Hash-based)**: SPHINCS+
3. **多變量密碼 (Multivariate)**
4. **碼密碼 (Code-based)**

---

#### (3) 後量子密碼遷移指引 (114年高考二等考題!)

**應涵蓋內容**:

**Phase 1: 盤點與評估 (Inventory & Assessment)**

```
【資產盤點】
識別所有使用密碼的系統:
├─ TLS/SSL憑證
├─ VPN
├─ 程式碼簽章
├─ 資料庫加密
└─ IoT設備

【風險評估】
優先級排序:
高風險: 長期保密資料 (如醫療紀錄)
中風險: 中期保密資料
低風險: 短期資料
```

**Phase 2: 測試與驗證 (Testing & Validation)**

```
【相容性測試】
├─ PQC演算法效能測試
├─ 既有系統相容性
└─ 硬體支援度

【試點部署】
選擇非關鍵系統先測試
```

**Phase 3: 混合模式 (Hybrid Approach)**

```
【為何需要混合?】
PQC演算法尚未經過時間考驗

【混合加密】
同時使用傳統 + PQC演算法
範例:
TLS連線 = ECDHE (傳統) + Kyber (PQC)
→ 即使其中一個被破解,仍安全
```

**Phase 4: 分階段遷移 (Phased Migration)**

```
【遷移優先順序】
第1階段: 外部facing系統 (TLS憑證)
第2階段: 內部關鍵系統 (資料庫加密)
第3階段: IoT設備
第4階段: Legacy系統

【時程規劃】
2024-2026: 試點與混合模式
2027-2030: 全面遷移
2030+: 純PQC
```

**Phase 5: 演算法敏捷性 (Crypto Agility)**

```
【設計原則】
系統設計要能「快速切換演算法」
避免寫死: AES-256
改用: 可配置的密碼模組

【為何重要?】
若PQC演算法被破解
→ 可快速切換到新演算法
```

**Phase 6: 教育訓練**

```
├─ 開發人員: PQC API使用
├─ 資安團隊: 漏洞評估
└─ 管理層: 風險認知
```

### 六、SBOM (Software Bill of Materials) - 113年高考新增!

#### (1) 供應鏈安全威脅

**問題**: 現代軟體高度依賴**第三方元件**

```
【例】一個網站程式:
├─ 自己寫的程式碼: 10%
└─ 第三方套件: 90%
    ├─ React框架
    ├─ Express.js
    ├─ 數百個npm套件
    └─ 每個套件又依賴其他套件...
```

**供應鏈攻擊案例**:

**案例1: Log4Shell (2021)**

```
【漏洞】
Log4j (Java日誌套件) 有遠端程式執行漏洞
CVSS 10.0/10 (最高危!)

【影響】
全球數百萬系統受影響
├─ Apple iCloud
├─ Tesla
├─ Steam
└─ 政府機關

【問題】
很多公司根本不知道自己用了Log4j!
→ 無法及時修補
```

**案例2: SolarWinds (2020)**

```
【攻擊】
駭客入侵SolarWinds公司
→ 在軟體更新中植入後門
→ 18,000個客戶下載惡意更新

【受害者】
├─ 美國財政部
├─ 國務院
├─ Microsoft
└─ ...

【損失】
數十億美元
```

**案例3: npm套件投毒**

```
【社交工程】
駭客假裝成套件維護者
→ 要求轉移維護權
→ 植入惡意程式碼

【範例】
event-stream套件被植入竊取比特幣的程式碼
→ 200萬次下載量
```

---

#### (2) SBOM 定義與重要性

**SBOM**: 軟體物料清單,列出軟體中**所有元件**的清單,類似「食品成分標示」。

**為何重要?**

```
【沒有SBOM】
Log4Shell爆發 → 公司問「我們有用Log4j嗎?」
→ 不知道! (深藏在第N層依賴)
→ 無法快速應對

【有SBOM】
Log4Shell爆發 → 查SBOM
→ 發現系統A、D有用Log4j 4.15版
→ 立即修補
```

---

#### (3) SBOM 內容

**必要資訊**:

```
【每個元件應記錄】
├─ 元件名稱: Log4j
├─ 版本: 2.14.1
├─ 供應商: Apache Software Foundation
├─ 授權: Apache License 2.0
├─ 依賴關係: 依賴A, B, C套件
└─ 雜湊值: SHA256校驗
```

**SBOM格式標準**:

1. **SPDX** (Software Package Data Exchange) - Linux Foundation
2. **CycloneDX** - OWASP
3. **SWID** (Software Identification Tags)

---

#### (4) SBOM 在供應鏈管理的應用

**應用1: 漏洞管理**

```
【流程】
1. 產生SBOM: 列出所有元件
2. 訂閱CVE通知
3. 新漏洞公布 → 查SBOM
4. 發現有受影響元件 → 修補

自動化:
SBOM + CVE資料庫 → 自動警報
```

**應用2: 授權合規**

```
【問題】
使用GPL授權套件 → 整個專案要開源

【SBOM解法】
查SBOM → 發現有GPL套件
→ 違反公司政策 → 刪除/替換
```

**應用3: 採購決策**

