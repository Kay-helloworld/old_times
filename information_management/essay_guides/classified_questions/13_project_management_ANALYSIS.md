## 📚 核心知識解析

### 一、專案管理基礎

#### (1) 專案 (Project) 定義

**專案**: 為創造**獨特的產品、服務或成果**而進行的**臨時性**工作。

**PMBOK定義**: A temporary endeavor undertaken to create a unique product, service, or result.

**三大核心特性**:

```
1. 臨時性 (Temporary)
   ├─ 有明確的起始與結束時間
   ├─ 不是永久性的
   └─ 範例: 開發新系統 (6個月)

2. 獨特性 (Unique)
   ├─ 每個專案都不同
   ├─ 即使重複專案,環境/人員都不同
   └─ 範例: 雖然都是開發網站,但每個客戶需求不同

3. 漸進明細 (Progressive Elaboration)
   ├─ 隨著專案進行,逐步細化
   ├─ 起始階段: 概括性計畫
   └─ 執行階段: 詳細計畫
```

---

**專案 vs 例行營運 (Operations)**

| 面向 | 專案 (Project) | 例行營運 (Operations) |
|------|---------------|---------------------|
| **時間** | 臨時性 (有起訖) | **持續進行** (ongoing) |
| **產出** | 獨特的 | **重複性的** |
| **目標** | 達成特定目標後結束 | **維持營運** |
| **變更** | 變更頻繁 | 標準化流程 |
| **人員** | 臨時組成團隊 | 固定組織 |
| **範例** | • 開發新系統<br>• 建造大樓<br>• 舉辦活動 | • 日常維運<br>• 客服支援<br>• 生產線 |

**專案與營運的關係**:
```
專案結束 → 移交給營運
範例:
├─ 專案: 開發客服系統 (6個月)
└─ 營運: 日常使用與維護系統
```

---

#### (2) 專案管理 (Project Management) 定義

**專案管理**: 將**知識、技能、工具與技術**應用於專案活動,以**滿足專案需求**。

**專案成功的衡量標準** (傳統三角限制):

```
        範疇 (Scope)
           /\
          /  \
         /    \
        /      \
       /        \
      /  專案成功 \
     /____________\
   時間          成本
  (Time)       (Cost)

【三角限制】
範疇↑ → 時間↑ 或 成本↑
時間↓ → 範疇↓ 或 成本↑
成本↓ → 範疇↓ 或 時間↑

平衡這三者 = 專案管理的藝術
```

**現代專案成功因素** (擴展到6個):
1. 範疇 (Scope)
2. 時間 (Time)
3. 成本 (Cost)
4. **品質 (Quality)**
5. **資源 (Resources)**
6. **風險 (Risk)**

---

#### (3) 專案利害關係人 (Stakeholders)

**利害關係人**: 專案會**影響**或被專案**影響**的個人/組織。

**主要利害關係人**:

| 利害關係人 | 角色與關注點 |
|-----------|------------|
| **專案發起人 (Sponsor)** | 提供資源、最高決策者<br>關注: 投資報酬率(ROI) |
| **專案經理 (PM)** | 負責專案成敗<br>關注: 範疇/時間/成本 |
| **專案團隊** | 執行專案工作<br>關注: 工作分配、資源 |
| **客戶/使用者** | 使用專案成果<br>關注: 需求滿足、品質 |
| **供應商** | 提供外部資源<br>關注: 合約、付款 |
| **PMO (專案管理辦公室)** | 提供專案管理標準<br>關注: 方法論一致性 |
| **高階管理層** | 策略層級決策<br>關注: 與組織策略一致性 |

**利害關係人管理**:

```
【識別】
列出所有利害關係人

【分析】
權力/利益矩陣:

        權力高
          ↑
  管理     |  密切管理
  (滿足)   |  (重點對象)
  ─────────┼─────────→ 利益高
  監督     |  告知
  (最少)   |  (保持滿意)
          ↓
        權力低

【規劃】
溝通計畫、參與策略

【管理】
維持關係、解決衝突
```

#### 流程1: 起始流程群組 (Initiating)

**目的**: **正式授權**專案或階段的開始

**主要活動**:

**1.1 制定專案章程 (Develop Project Charter)**

**專案章程**: 正式授權專案存在的文件,賦予專案經理動用資源的權力。

**專案章程內容**:

```

│      專案章程 (Project Charter)     │
├─────────────────────────────┤
│ 1. 專案目的/理由               │
│    - 商業需求                  │
│    - 專案目標 (SMART)          │
│                                │
│ 2. 高階需求                    │
│    - 主要可交付成果            │
│    - 專案範疇概述              │
│                                │
│ 3. 專案經理指派                │
│    - 專案經理姓名              │
│    - 權責範圍                  │
│                                │
│ 4. 時程里程碑                  │
│    - 主要階段與日期            │
│                                │
│ 5. 預算估算                    │
│    - 概估成本                  │
│                                │
│ 6. 假設與限制                  │
│    - 假設條件                  │
│    - 已知限制                  │
│                                │
│ 7. 風險摘要                    │
│    - 高階風險列表              │
│                                │
│ 8. 利害關係人                  │
│    - 主要利害關係人列表        │
│                                │
│ 9. 授權簽名                    │
│    - 發起人簽名                │
│    - 授權日期                  │
└─────────────────────────────┘
```

**專案目標 SMART 原則**:

```
S - Specific (具體明確)
    ❌ "改善系統效能"
    ✅ "將系統回應時間從3秒降到1秒以內"

M - Measurable (可衡量)
    ✅ 包含數字、指標

A - Achievable (可達成)
    ✅ 考量資源與限制,實際可行

R - Relevant (相關性)
    ✅ 與組織策略一致

T - Time-bound (有時限)
    ✅ "於2024年12月31日前完成"
```

#### 流程2: 規劃流程群組 (Planning)

**目的**: 定義**專案範疇**,精細化**目標**,制定達成目標的**行動方案**。

**核心理念**: "Failing to plan is planning to fail" (沒有規劃就是規劃失敗)

**主要規劃子流程** (共24個,列舉核心):

---

**2.1 規劃範疇管理**

**工具: WBS (Work Breakdown Structure, 工作分解結構)**

**定義**: 將專案**逐層分解**成更小、更易管理的**工作包**。

**WBS 範例: 開發網站專案**

```
1.0 網站開發專案
│
├─ 1.1 專案管理
│   ├─ 1.1.1 專案規劃
│   ├─ 1.1.2 進度監控
│   └─ 1.1.3 結案報告
│
├─ 1.2 需求分析
│   ├─ 1.2.1 訪談使用者
│   ├─ 1.2.2 需求文件撰寫
│   └─ 1.2.3 需求審查
│
├─ 1.3 系統設計
│   ├─ 1.3.1 架構設計
│   ├─ 1.3.2 資料庫設計
│   ├─ 1.3.3 UI/UX設計
│   └─ 1.3.4 設計審查
│
├─ 1.4 系統開發
│   ├─ 1.4.1 前端開發
│   ├─ 1.4.2 後端開發
│   ├─ 1.4.3 資料庫建置
│   └─ 1.4.4 整合
│
├─ 1.5 測試
│   ├─ 1.5.1 單元測試
│   ├─ 1.5.2 整合測試
│   ├─ 1.5.3 系統測試
│   └─ 1.5.4 使用者驗收測試(UAT)
│
└─ 1.6 部署與上線
    ├─ 1.6.1 環境準備
    ├─ 1.6.2 系統部署
    ├─ 1.6.3 教育訓練
    └─ 1.6.4 正式上線
```

**WBS 編碼系統**:
- 1.0 = 專案
- 1.1 = 第一層級
- 1.1.1 = 第二層級 (工作包)

**WBS 分解原則**:

```
【100% 規則】
每一層級的子項目加總 = 100%父項目
避免遺漏或重複

【8/80 規則】
工作包的工時: 8小時 ~ 80小時
├─ 太小 (< 8hr): 管理成本高
└─ 太大 (> 80hr): 難以掌控

【可交付成果導向】
以「產出」而非「活動」命名
✅ "需求文件"
❌ "撰寫需求"
```

**2.2 規劃時程管理**

**工具1: 甘特圖 (Gantt Chart)**

```
任務        1月  2月  3月  4月  5月  6月
需求分析    ████
系統設計      ████████
系統開發          ████████████
測試                    ████████
部署                        ████

█ = 工作時間
```

**工具2: PERT/CPM (要徑法)**

**定義**: 找出專案的**關鍵路徑** (Critical Path),即決定專案最短完成時間的路徑。

**範例**:

```
    A(5天)
   ╱      ╲
Start      → D(3天) → End
   ╲      ╱
    B(2天)→C(4天)

路徑1: S → A → D → E = 5+3 = 8天
路徑2: S → B → C → D → E = 2+4+3 = 9天 ← 關鍵路徑!

關鍵路徑 = 9天 (專案最短完成時間)
路徑2上的活動 = 關鍵活動 (延遲會影響整個專案)
```

**浮時 (Float/Slack)**:
```
路徑1的浮時 = 9 - 8 = 1天
意義: 活動A可延遲1天而不影響專案
```

**2.3 規劃成本管理**

**成本估算技術**:

| 技術 | 說明 | 準確度 |
|------|------|--------|
| **類比估算** | 參考類似專案 | ±50% |
| **參數估算** | 根據參數計算<br>(如:每功能點$5000) | ±25% |
| **三點估算** | (樂觀+4×最可能+悲觀)/6 | ±10% |
| **由下而上估算** | 加總所有工作包 | ±5% |

**預算範例**:

```
WBS          估算成本   應變準備金   管理準備金
1.1 需求分析  $50,000
1.2 系統設計  $100,000
1.3 開發      $300,000
1.4 測試      $80,000
1.5 部署      $40,000
───────────────────────────────────
小計          $570,000
應變準備金    $57,000  (10%, 已知風險)
───────────────────────────────────
成本基線      $627,000
管理準備金    $63,000  (10%, 未知風險)
───────────────────────────────────
總預算        $690,000
```

#### 流程3: 執行流程群組 (Executing)

**目的**: **執行專案管理計畫**中定義的工作,完成專案目標。

**特性**: 消耗**最多資源** (時間/人力/成本)的階段,約佔整個專案的**60-70%**。

**主要活動**:

**3.1 指導與管理專案工作**

```
【執行內容】
├─ 執行計畫中的活動
├─ 產生可交付成果 (Deliverables)
├─ 管理變更
└─ 蒐集工作績效資料

【可交付成果】
專案產出的有形或無形產品
範例:
├─ 需求規格書 (文件)
├─ 系統原型 (軟體)
├─ 測試報告 (文件)
└─ 正式上線系統 (產品)
```

**3.2 管理專案團隊**

```
【團隊發展階段】(Tuckman模型)

1. Forming (組建期)
   ├─ 團隊剛組成,互相認識
   └─ PM角色: 建立目標、角色分工

2. Storming (風暴期)
   ├─ 開始出現衝突、意見不合
   └─ PM角色: 衝突管理、建立規範

3. Norming (規範期)
   ├─ 建立共識、團隊凝聚力↑
   └─ PM角色: 促進合作

4. Performing (績效期)
   ├─ 團隊高效運作
   └─ PM角色: 授權、支援

5. Adjourning (解散期)
   ├─ 專案結束,團隊解散
   └─ PM角色: 肯定成果、移交
```

**3.3 管理溝通**

```
【溝通公式】
n個人的溝通管道數 = n(n-1)/2

範例:
5人團隊 = 5×4/2 = 10條溝通管道
10人團隊 = 10×9/2 = 45條! (成長快速)

啟示: 團隊越大,溝通成本越高
```

| **可交付成果** | 專案產出的產品/服務/成果 |

#### 流程4: 監控流程群組 (Monitoring & Controlling)

**目的**: **追蹤、審查、調整**專案進度與績效,**識別需要變更**的地方。

**特性**: 與**執行流程並行**,貫穿整個專案。

**主要活動**:

**4.1 監控專案工作**

**工具: 掙值管理 (EVM, Earned Value Management)**

**核心概念**: 整合**範疇、時程、成本**,提供專案績效的客觀度量。

**三大基礎值**:

```
PV (Planned Value, 計畫值)
= 應該完成的工作的預算價值
= "計畫要花多少錢?"

EV (Earned Value, 掙值)
= 實際完成的工作的預算價值
= "實際完成的工作值多少錢?"

AC (Actual Cost, 實際成本)
= 實際完成工作所花的成本
= "實際花了多少錢?"
```

**績效指標**:

```
【成本績效指標 CPI】
CPI = EV / AC

CPI > 1 → 成本節省 (花得少,做得多)
CPI = 1 → 符合預算
CPI < 1 → 成本超支 (花得多,做得少)

【時程績效指標 SPI】
SPI = EV / PV

SPI > 1 → 進度超前 (做得比計畫多)
SPI = 1 → 符合進度
SPI < 1 → 進度落後 (做得比計畫少)
```

**實例計算**:

```
【範例】
專案預算: $100,000, 工期: 10週
目前: 第5週結束

PV (計畫到第5週應完成) = $50,000
EV (實際完成的工作價值) = $40,000
AC (實際花費) = $45,000

【分析】
CPI = 40,000 / 45,000 = 0.89 < 1
    → 成本超支 (花$45K才做出$40K的工作)

SPI = 40,000 / 50,000 = 0.80 < 1
    → 進度落後 (應完成$50K,只完成$40K)

【結論】
專案同時面臨成本超支與進度落後!
需要採取糾正措施
```

**預測**:

```
EAC (Estimate at Completion, 完工估算)
= BAC / CPI
= $100,000 / 0.89
= $112,360

意義: 若照目前績效,專案完成時會花$112,360
      超出預算$12,360!
```

### 三、風險管理 (Risk Management) - 113年地特完整考題!

#### (1) 風險定義

**風險 (Risk)**: **不確定事件或條件**,一旦發生,會對專案目標產生**正面或負面**影響。

**風險 vs 問題**:

| 面向 | 風險 (Risk) | 問題 (Issue) |
|------|------------|-------------|
| 時間 | **未來可能發生** | **已經發生** |
| 確定性 | 不確定 (機率<100%) | 確定 (已發生) |
| 處理 | **風險管理** (預防) | **問題管理** (解決) |
| 範例 | "關鍵人員**可能**離職" | "關鍵人員**已經**離職" |

**風險特性**:
```
機率 (Probability): 發生的可能性 (0-100%)
影響 (Impact): 對目標的影響程度
風險程度 = 機率 × 影響
```

---

#### (2) 風險管理4步驟 (完整流程)

**步驟1: 風險辨識 (Risk Identification)**

**目的**: **盡早、持續**識別所有可能風險

**風險辨識方法**:

**方法1: 腦力激盪 (Brainstorming)**
```
【程序】
1. 召集專案團隊+專家
2. 自由發想,列舉所有可能風險
3. 不批評、不限制
4. 記錄所有意見
5. 最後整理分類

【優點】
✅ 快速、簡單
✅ 團隊參與,凝聚共識

【缺點】
❌ 可能受群體思考影響
❌ 主觀性強
```

**方法2: 德爾菲法 (Delphi Technique)**
```
【程序】
1. 匿名問卷:專家各自列風險
2. 彙整結果,再次發問卷
3. 專家參考他人意見,調整答案
4. 重複3-4輪,直到達成共識

【優點】
✅ 匿名,避免權威影響
✅ 專家意見

【缺點】
❌ 耗時
❌ 需要專家資源
```

**方法3: 檢查表 (Checklist)**
```
【程序】
參考類似專案的風險清單
逐項檢查是否適用

【優點】
✅ 快速、不遺漏常見風險

【缺點】
❌ 可能遺漏新型態風險
```

**方法4: SWOT分析**
```
【程序】
分析專案的優勢/劣勢/機會/威脅
從 Weakness & Threat 產生風險

範例:
Weakness: 團隊缺乏XX技術經驗
→ 風險: 技術問題可能延遲進度
```

**方法5: 假設分析 (Assumption Analysis)**
```
【程序】
列出專案所有假設
挑戰每個假設的有效性
假設不成立 → 產生風險

範例:
假設: "關鍵供應商會準時交貨"
風險: "供應商可能延遲交貨"
```

---

**風險分類** (RBS, Risk Breakdown Structure):

```
專案風險
│
├─ 技術風險
│   ├─ 技術不成熟
│   ├─ 系統整合困難
│   └─ 效能無法達標
│
├─ 管理風險
│   ├─ 範疇蔓延 (Scope Creep)
│   ├─ 溝通不良
│   └─ 決策延遲
│
├─ 組織風險
│   ├─ 資源不足
│   ├─ 優先級改變
│   └─ 組織重組
│
├─ 外部風險
│   ├─ 法規改變
│   ├─ 市場變化
│   └─ 天災
│
└─ 專案管理風險
    ├─ 估算不準
    ├─ 計畫不周延
    └─ 控制不足
```

---

**步驟2: 風險分析 (Risk Analysis)**

**目的**: 評估風險的**機率**與**影響**

**定性分析 (Qualitative Analysis)**:

**機率評級**:
```
很高 (>70%)   = 5
高 (50-70%)   = 4
中 (30-50%)   = 3
低 (10-30%)   = 2
很低 (<10%)   = 1
```

**影響評級**:

| 影響 | 成本 | 時程 | 範疇 | 品質 | 評分 |
|------|------|------|------|------|------|
| **很高** | 超支>40% | 延遲>20% | 無法使用 | 嚴重缺陷 | 5 |
| **高** | 超支20-40% | 延遲10-20% | 主要功能受影響 | 明顯缺陷 | 4 |
| **中** | 超支10-20% | 延遲5-10% | 次要功能受影響 | 有缺陷 | 3 |
| **低** | 超支<10% | 延遲<5% | 小幅影響 | 輕微缺陷 | 2 |
| **很低** | 可忽略 | 可忽略 | 幾乎無影響 | 幾乎無影響 | 1 |

**風險評分**:
```
風險評分 = 機率 × 影響

範例:
風險: 關鍵人員離職
機率: 中 (3)
影響: 高 (4)
風險評分 = 3 × 4 = 12
```

---

**定量分析 (Quantitative Analysis)**:

**EMV (Expected Monetary Value, 期望貨幣價值)**

```
EMV = 機率 × 影響金額

【範例1: 負面風險】
風險: 供應商延遲交貨
機率: 30%
影響: 延遲2週,損失$50,000
EMV = 0.3 × (-$50,000) = -$15,000

【範例2: 正面風險(機會)】
機會: 提早完成獲得獎金
機率: 20%
影響: 獎金$30,000
EMV = 0.2 × $30,000 = $6,000

【決策】
應變基金需準備: $15,000 (應付負面風險)
```

**決策樹分析**:

```
【情境】
開發新功能 vs 購買現成模組

決策點
├─ 選項A: 自行開發 ($80,000)
│   ├─ 成功 (70%): 省$20,000
│   │   EMV = 0.7 × $20,000 = $14,000
│   └─ 失敗 (30%): 損失$50,000
│       EMV = 0.3 × (-$50,000) = -$15,000
│   總EMV = $14,000 - $15,000 = -$1,000
│
└─ 選項B: 購買模組 ($100,000)
    └─ 確定成功 (100%)
        EMV = $0

【決策】
選項B (購買) EMV較高 ($0 > -$1,000)
→ 建議購買現成模組
```

---

**步驟3: 風險優先順序排序 (Risk Prioritization)**

**工具: 風險矩陣 (Risk Matrix)**

**3×3 矩陣**:

```
        影響 (Impact)
機率    低    中    高
(P)     1     2     3
高 3    M     H     H
中 2    L     M     H
低 1    L     L     M

L = 低優先級 (接受)
M = 中優先級 (監控)
H = 高優先級 (立即處理)
```

**5×5 矩陣** (更精細):

```
        影響 →
機率    1    2    3    4    5
↓
5      M    H    H    VH   VH
4      M    M    H    H    VH
3      L    M    M    H    H
2      L    L    M    M    H
1      L    L    L    M    M

L = 低 (1-5)
M = 中 (6-12)
H = 高 (13-20)
VH = 非常高 (21-25)
```

**風險登記簿 (Risk Register)**:

```
┌────┬──────────┬────┬────┬──────┬────┬────────┐
│ID  │風險描述  │機率│影響│風險值│優先│負責人  │
├────┼──────────┼────┼────┼──────┼────┼────────┤
│R001│關鍵人員  │  3 │  4 │  12  │ H  │PM      │
│    │離職      │    │    │      │    │        │
├────┼──────────┼────┼────┼──────┼────┼────────┤
│R002│需求變更  │  4 │  3 │  12  │ H  │需求分析│
├────┼──────────┼────┼────┼──────┼────┼────────┤
│R003│硬體故障  │  2 │  3 │   6  │ M  │IT經理  │
└────┴──────────┴────┴────┴──────┴────┴────────┘
```

**排序原則**:
1. 先處理**高優先級**風險
2. 考慮**多重影響** (同時影響時程+成本)
3. 考慮**關聯性** (某風險觸發其他風險)

---

**步驟4: 風險處置 (Risk Treatment/Response)**

**4種基本策略**:

**策略1: 規避 (Avoidance)** - 消除威脅

```
【定義】
改變專案計畫,完全消除風險

【方法】
├─ 改變專案範疇
├─ 改變時程
├─ 改變策略
└─ 降低範疇

【範例】
風險: 新技術不成熟,可能失敗
規避: 改用成熟技術

風險: 颱風季可能影響戶外活動
規避: 改在室內舉辦

風險: 依賴單一供應商
規避: 改用雙供應商

【適用時機】
✅ 高機率 + 高影響
✅ 有可行的替代方案
✅ 規避成本可接受
```

---

**策略2: 緩解/降低 (Mitigation)** - 降低機率或影響

```
【定義】
採取措施降低風險發生機率或影響程度

【方法】
降低機率:
├─ 提早行動
├─ 增加測試
├─ 使用成熟技術
└─ 培訓團隊

降低影響:
├─ 建立備援
├─ 增加緩衝
└─ 分階段實施

【範例1: 降低機率】
風險: 關鍵人員離職 (機率30%)
緩解: 
├─ 知識移轉文件化
├─ 培養第二人才
├─ 改善工作環境
→ 降低機率到10%

【範例2: 降低影響】
風險: 硬體故障導致停機 (影響:損失$100K)
緩解:
├─ 建立備援系統 (HA)
├─ 定期備份
└─ 快速復原程序
→ 降低影響到$10K

【範例3: 軟體專案】
風險: 整合測試失敗
緩解:
├─ 提早進行單元測試
├─ 持續整合 (CI)
├─ 定期整合測試
→ 降低失敗機率

【適用時機】
✅ 無法完全規避
✅ 可透過行動降低風險
✅ 緩解成本 < 風險成本
```

---

**策略3: 轉移 (Transfer)** - 轉移給第三方

```
【定義】
將風險的影響轉移給第三方
(注意: 責任可能仍在)

【方法】
├─ 購買保險
├─ 外包
├─ 擔保/保證金
└─ 合約條款

【範例1: 保險】
風險: 設備損壞
轉移: 購買設備保險
→ 損失由保險公司承擔

【範例2: 外包】
風險: 缺乏XX專業技能
轉移: 外包給專業廠商
→ 技術風險轉移給廠商

【範例3: 固定價格合約】
風險: 成本超支
轉移: 簽訂固定價格合約
→ 超支風險由承包商承擔

【範例4: 罰則條款】
風險: 供應商延遲交貨
轉移: 合約訂定罰則
→ 延遲損失由供應商賠償

【適用時機】
✅ 有第三方願意承擔
✅ 轉移成本可接受
✅ 無法自行管理的風險 (如天災)

【注意】
⚠️ 轉移風險 ≠ 消除風險
⚠️ 仍需監控第三方表現
```

---

**策略4: 接受 (Acceptance)** - 承認並準備應變

```
【定義】
承認風險存在,不主動處理
(但要準備應變計畫)

【兩種類型】
主動接受: 建立應變基金/計畫
被動接受: 什麼都不做,等發生再說

【範例1: 主動接受】
風險: 多種小型風險
接受: 建立應變基金 (10%預算)
→ 風險發生時有資金應對

【範例2: 被動接受】
風險: 極低機率的颱風
接受: 不採取行動
→ 發生時再臨機應變

【適用時機】
✅ 低機率 + 低影響
✅ 無法經濟地處理
✅ 其他策略成本太高

【應變計畫 (Contingency Plan)】
IF (風險發生) THEN (執行應變計畫)

範例:
風險: 關鍵人員離職
應變: 立即啟用備援人員
      調整進度計畫
```

---

**正面風險(機會)的策略**:

| 威脅策略 | 機會策略 | 說明 |
|---------|---------|------|
| 規避 | **開拓 (Exploit)** | 確保機會發生 |
| 緩解 | **提高 (Enhance)** | 增加機會發生機率/影響 |
| 轉移 | **分享 (Share)** | 與第三方分享機會 |
| 接受 | **接受 (Accept)** | 不主動追求,但樂見發生 |

**範例**:
```
機會: 提早完成獲得獎金

開拓: 投入更多資源,確保提早完成
提高: 採用新工具,增加提早完成機率
分享: 與夥伴合作,共享獎金
接受: 正常執行,若提早完成就賺到
```

---

**風險處置決策矩陣**:

| 風險等級 | 建議策略 | 說明 |
|---------|---------|------|
| **VH (非常高)** | 規避 或 大幅緩解 | 無法接受,必須處理 |
| **H (高)** | 緩解 + 應變計畫 | 積極降低 |
| **M (中)** | 緩解 或 轉移 | 視成本效益決定 |
| **L (低)** | 接受 + 監控 | 建立應變基金 |

---

#### (3) 風險監控與控制

**持續活動**:

```
1. 追蹤已識別風險
   └─ 檢查風險狀態是否改變

2. 監控殘餘風險
   └─ 處置後仍存在的風險

3. 識別新風險
   └─ 持續進行風險辨識

4. 執行風險回應計畫
   └─ 觸發條件滿足時執行

5. 評估風險處理成效
   └─ 策略是否有效?
```

**風險監控會議** (定期):
- 審查風險登記簿
- 更新風險狀態
- 識別新風險
- 調整風險處置

### 四、SDLC 軟體開發方法論比較 (109年地特完整題!)

#### (1) SDLC 定義

**SDLC (Software Development Life Cycle)**: 軟體從**概念**到**退役**的完整生命週期。

**典型階段**:
```
1. 需求分析 → 2. 設計 → 3. 開發 → 4. 測試 → 5. 部署 → 6. 維護
```

---

#### (2) 四大開發方法論比較

**方法1: 瀑布式 (Waterfall)**

```
【流程】線性、順序執行

需求分析 ──→ 完成需求規格
    ↓
系統設計 ──→ 完成設計文件
    ↓
程式開發 ──→ 完成程式碼
    ↓
系統測試 ──→ 通過測試
    ↓
部署維護 ──→ 正式上線

【特點】
✅ 結構化、文件完整
✅ 易於管理與追蹤
✅ 適合需求明確的專案

❌ 不易應對變更
❌ 客戶要到最後才看到成果
❌ 風險發現晚

【適用情境】
✅ 需求明確且穩定
✅ 技術成熟
✅ 法規要求完整文件
✅ 範例: 政府系統、航太軍事
```

---

**方法2: 螺旋式 (Spiral)**

```
【流程】風險驅動的迭代模型

每個螺旋包含4個階段:
1. 計畫 → 2. 風險分析 → 3. 工程開發 → 4. 客戶評估
                ↓
            (重複螺旋,逐步完善)

【特點】
✅ 強調風險管理
✅ 結合瀑布+原型
✅ 中大型專案適用

❌ 複雜度高
❌ 需要風險管理專業
❌ 成本較高

【適用情境】
✅ 大型、高風險專案
✅ 創新性專案
✅ 需求不完全確定
✅ 範例: 新技術研發
```

---

**方法3: 敏捷式 (Agile)**

```
【流程】迭代、增量、快速交付

Sprint 1 (2週)  → 可用功能1
Sprint 2 (2週)  → 可用功能2
Sprint 3 (2週)  → 可用功能3
...
每個Sprint:
計畫 → 開發 → 測試 → 展示 → 回顧

【核心價值 (Agile Manifesto)】
✅ 個人與互動 > 流程與工具
✅ 可用軟體 > 詳盡文件
✅ 客戶合作 > 合約協商
✅ 回應變更 > 遵循計畫

【特點】
✅ 快速回應變更
✅ 頻繁交付價值
✅ 客戶持續參與
✅ 團隊自組織

❌ 文件較少
❌ 需要客戶高度參與
❌ 難以預測最終成本/時程

【適用情境】
✅ 需求不明確或快速變化
✅ 創新產品
✅ 客戶願意積極參與
✅ 範例: 新創產品、行動App
```

---

**方法4: DevOps**

```
【流程】開發(Dev)+運維(Ops)整合

持續整合 → 持續測試 → 持續部署 → 持續監控
     ↓          ↓          ↓           ↓
   自動化      自動化      自動化       即時回饋

【特點】
✅ 自動化 (CI/CD)
✅ 快速部署 (一天多次)
✅ Dev與Ops協作
✅ 監控與回饋

❌ 需要文化轉變
❌ 初期投資高 (工具)
❌ 需要高技術能力

【適用情境】
✅ 需頻繁更新的系統
✅ 雲端原生應用
✅ 微服務架構
✅ 範例: SaaS服務
```

---

**四種方法比較總表**:

| 面向 | 瀑布式 | 螺旋式 | 敏捷式 | DevOps |
|------|--------|--------|--------|--------|
| **彈性** | 低 | 中 | 高 | 高 |
| **文件** | 完整 | 完整 | 精簡 | 精簡 |
| **風險管理** | 後期 | **每階段** | 每Sprint | 持續 |
| **客戶參與** | 少 | 中 | **頻繁** | 頻繁 |
| **交付頻率** | 最後 | 每螺旋 | 每2週 | **每日** |
| **適合專案規模** | 大中小 | **大型** | 中小 | 中小 |
| **成本可預測性** | 高 | 中 | 低 | 中 |

## 🎯 申論題答題技巧總結

### 技巧一: 5大流程必背口訣

**記憶口訣**: 「**起規執監結**」
1. **起**始 (Initiating)
2. **規**劃 (Planning)
3. **執**行 (Executing)
4. **監**控 (Monitoring & Controlling)
5. **結**案 (Closing)

**標準答題架構** (20-25分題):
```
時間分配:
1. 定義 (2分鐘) → 3分
2. 5大流程 (15分鐘) → 15分
   每個流程:
   ├─ 流程名稱 (中英文)
   ├─ 主要工作 (3-4項)
   └─ 產出文件 (1-2項)
3. 流程關係圖 (2分鐘) → 2分
```

**必畫圖**: 流程關係圖
```
  起始 ──→ 規劃 ──→ 執行
   ↓        ↓       ↓
   └────→ 監控 ←────┘
             ↓
           結案
```

**加分技巧**:
- ✅ 提及**PMBOK** (PMI標準)
- ✅ 說明**專案三角** (範疇/時間/成本)
- ✅ 舉例: "如開發網站專案..."

### 技巧二: 風險管理4步驟萬用架構

**記憶口訣**: 「**辨分排處**」
1. **辨**識 (Identification)
2. **分**析 (Analysis)
3. **排**序 (Prioritization)
4. **處**置 (Treatment)

**每個步驟必答內容**:

**步驟1: 辨識**
- 方法 (至少3個): 腦力激盪、德爾菲、檢查表
- 風險分類: 技術/管理/組織/外部

**步驟2: 分析**
- 定性: 機率×影響評級
- 定量: EMV計算公式

**步驟3: 排序**
- **必畫**: 風險矩陣 (3×3)
```
     影響
機率 低 中 高
高   M  H  H
中   L  M  H
低   L  L  M
```

**步驟4: 處置**
- **4種策略**: 規避/緩解/轉移/接受
- **必說明**: 每種策略的**適用時機**

**範例說明**:
```
風險: 關鍵人員離職
辨識: 透過腦力激盪識別
分析: 機率中(3)×影響高(4)=12
排序: 高優先級 (H)
處置: 緩解 (培養第二人才、知識移轉)
```

### 技巧五: 掙值管理EVM計算題

**必背公式**:

```
【基礎】
PV = 計畫值 (應該完成的工作預算)
EV = 掙值 (實際完成的工作預算價值)
AC = 實際成本 (實際花費)

【績效指標】
CPI = EV / AC  (成本績效指標)
  > 1 → 省錢 ✅
  < 1 → 超支 ❌

SPI = EV / PV  (時程績效指標)
  > 1 → 超前 ✅
  < 1 → 落後 ❌

【預測】
EAC = BAC / CPI (完工估算)
ETC = EAC - AC (尚需成本)
VAC = BAC - EAC (完工差異)
```

**計算步驟**:
1. 找出PV, EV, AC
2. 計算CPI, SPI
3. 判斷專案狀態
4. 預測完工成本(EAC)

### 技巧六: 常見陷阱避免

**陷阱1**: 混淆專案與營運
- ❌ "系統維護是專案"
- ✅ 正確: 系統維護是**營運**,系統開發才是**專案**

**陷阱2**: 混淆風險與問題
- ❌ "關鍵人員已離職"是風險
- ✅ 正確: 已發生=**問題**, 可能發生=**風險**

**陷阱3**: 風險處置策略用錯
- ❌ "低機率低影響的風險要規避"
- ✅ 正確: 低機率低影響 → **接受**

**陷阱4**: WBS分解錯誤
- ❌ 以"活動"命名 (如"撰寫文件")
- ✅ 正確: 以**可交付成果**命名 (如"需求文件")

**陷阱5**: SDLC方法選錯
- ❌ "政府系統用敏捷式"
- ✅ 正確: 政府系統需求明確、需完整文件 → **瀑布式**
