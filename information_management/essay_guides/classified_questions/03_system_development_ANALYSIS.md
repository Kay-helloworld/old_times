## 📚 核心知識解析

### 一、SDLC vs SSDLC 完整比較

| 項目 | 傳統SDLC | 安全SSDLC |
|------|---------|-----------|
| **關注焦點** | 功能需求、時程、成本 | **功能需求 + 資安需求** |
| **需求階段** | 蒐集功能需求 | **+ 威脅建模、資安需求分析** |
| **設計階段** | 系統架構設計 | **+ 安全架構設計、最小權限原則** |
| **實作階段** | 程式撰寫 | **+ 安全編碼規範、輸入驗證** |
| **測試階段** | 功能測試 | **+ 弱點掃描、滲透測試、OWASP檢測** |
| **維護階段** | 錯誤修正、功能擴充 | **+ 安全patch管理、事件應變** |
| **成本** | 較低 | 較高 (但降低資安事件損失) |
| **適用情境** | 一般系統 | **高敏感系統、金融、醫療、政府** |

**SSDLC五大階段 (詳細版)**:

```
階段1: 需求分析 (Requirements)
├─ 識別資產與威脅
├─ 定義安全需求 (CIA要求)
├─ 威脅建模 (STRIDE、DREAD)
└─ 合規要求確認 (GDPR、個資法)

階段2: 設計 (Design)
├─ 安全架構設計
├─ 最小權限原則
├─ 資料加密設計
├─ 身分驗證機制
└─ 存取控制設計

階段3: 實作 (Implementation)
├─ 安全編碼規範 (OWASP Secure Coding)
├─ 輸入驗證與消毒
├─ 輸出編碼
├─ 錯誤處理 (不洩漏敏感資訊)
└─ 程式碼審查 (Code Review)

階段4: 測試/驗證 (Verification)
├─ 靜態分析 (SAST) - 分析原始碼
├─ 動態分析 (DAST) - 執行時測試
├─ 弱點掃描 (Vulnerability Scanning)
├─ 滲透測試 (Penetration Testing)
└─ OWASP Top 10 檢測

階段5: 維護/部署 (Maintenance)
├─ 安全patch管理
├─ 事件監控與日誌
├─ 事件應變計畫
├─ 定期安全審查
└─ 持續改進
```

### 二、SDLC開發方法論完整比較

#### 1. 瀑布式 (Waterfall)

**定義**: 順序性的開發模式,各階段依序完成,前一階段完成才能進入下一階段。

**流程**:
```
需求分析 → 系統分析 → 系統設計 → 程式撰寫 → 測試 → 維護
   ↓          ↓          ↓          ↓        ↓       ↓
 完成後才能進入下一階段 (嚴格順序)
```

**特性**:
- ✅ 結構化、文件完整
- ✅ 每階段有明確交付物
- ✅ 易於管理與追蹤進度
- ❌ 缺乏彈性,需求變更成本高
- ❌ 晚期才看到成品,風險高

**適用情境**:
- 需求明確且穩定
- 專案規模小至中型
- 法規要求嚴格 (需完整文件)
- 傳統政府專案

**為何瀑布式要分階段?**
1. **風險控制**: 每階段檢查點降低風險
2. **品質保證**: 階段審查確保品質
3. **資源管理**: 分階段投入資源
4. **責任明確**: 各階段負責人清楚
5. **文件完整**: 便於後續維護

---

#### 2. 敏捷式 (Agile)

**定義**: 迭代式、增量式開發,強調快速交付可用軟體,持續回饋與調整。

**核心價值 (敏捷宣言 Agile Manifesto)**:
1. **個人與互動** 重於 流程與工具
2. **可用的軟體** 重於 詳盡的文件
3. **與客戶合作** 重於 合約協商
4. **回應變化** 重於 遵循計畫

**敏捷12原則** (核心重點):
- 透過及早和持續交付有價值的軟體來滿足客戶
- 歡迎需求變更,即使在開發後期
- 頻繁交付可用軟體 (每2週到2個月)
- 業務人員與開發者每天合作
- 面對面溝通最有效
- 可用軟體是進度的首要度量
- 持續關注技術卓越與良好設計

**流程** (以Scrum為例):
```
Product Backlog (產品待辦清單)
     ↓
Sprint Planning (衝刺規劃) - 選擇本次Sprint要完成的項目
     ↓
Sprint (1-4週) - 每日站立會議 (Daily Scrum)
     ↓
Sprint Review (展示成果給客戶)
     ↓
Sprint Retrospective (團隊回顧改進)
     ↓
交付可用軟體 → 下一個Sprint
```

**特性**:
- ✅ 快速適應需求變更
- ✅ 頻繁交付,及早發現問題
- ✅ 客戶參與度高
- ❌ 文件較少,不利後續維護
- ❌ 需要高度自律團隊
- ❌ 難以預估總成本與時程

**適用情境**:
- 需求不明確或快速變化
- 新創公司、網路服務
- 需要快速上市 (Time to Market)
- 小型敏捷團隊 (5-9人)

---

#### 3. 螺旋式 (Spiral)

**定義**: 結合瀑布式與雛型法,強調**風險分析**,每輪迭代包含風險評估。

**流程** (四象限循環):
```
第1輪迭代:
  ┌─────────────────────┐
  │ 1.計畫 (Planning)   │
  │   確定目標          │
  └──────┬──────────────┘
         ↓
  ┌──────────────────────┐
  │ 2.風險分析 (Risk)    │
  │   識別風險、雛型     │
  └──────┬───────────────┘
         ↓
  ┌──────────────────────┐
  │ 3.工程 (Engineering) │
  │   開發、測試         │
  └──────┬───────────────┘
         ↓
  ┌──────────────────────┐
  │ 4.評估 (Evaluation)  │
  │   客戶審查          │
  └──────┬───────────────┘
         ↓
      第2輪迭代...
```

**特性**:
- ✅ 強調風險管理 (每輪都評估)
- ✅ 適合大型複雜專案
- ✅ 彈性高,可容納需求變更
- ❌ 成本高 (需風險專家)
- ❌ 複雜度高,管理困難
- ❌ 不適合小型專案

**適用情境**:
- 高風險、高成本專案
- 技術不確定性高
- 大型企業系統
- 國防、航太專案

---

#### 4. DevOps

**定義**: Development + Operations,打破開發與維運部門的藩籬,強調**持續整合(CI)、持續交付(CD)**。

**核心理念**:
```
開發 (Dev) ←─── 緊密協作 ───→ 維運 (Ops)
     ↓                            ↓
  快速開發                      穩定營運
     ↓                            ↓
         持續整合/持續交付 (CI/CD)
```

**CI/CD流程**:
```
程式碼提交 → 自動建置 → 自動測試 → 自動部署
    (Git)      (Build)    (Test)     (Deploy)
                  ↓          ↓           ↓
               Jenkins   Unit Test   Docker容器
                         整合測試    K8s部署
```

**特性**:
- ✅ 快速部署 (每天多次)
- ✅ 自動化程度高
- ✅ 快速回饋,問題快速修復
- ❌ 需要完善自動化工具
- ❌ 文化轉型困難
- ❌ 初期建置成本高

**適用情境**:
- 網路服務、SaaS產品
- 需要頻繁更新
- 微服務架構
- 雲端原生應用

---

#### 5. 雛型法 (Prototyping)

**定義**: 先快速建立系統雛型展示給使用者,根據回饋修正,最終開發完整系統。

**流程**:
```
識別基本需求 → 建立初步雛型 → 使用者評估
                     ↓              ↓
                修正雛型 ←─── 收集回饋
                     ↓
           滿意後轉為正式系統
```

**特性**:
- ✅ 及早驗證需求
- ✅ 使用者參與度高
- ✅ 降低需求誤解風險
- ❌ 可能無限修改
- ❌ 雛型品質可能不佳
- ❌ 使用者期望過高

**適用時機**:
- 使用者需求不明確
- 使用者介面重要專案
- 創新應用 (無前例可循)

---

#### 6. 漸增式 (Incremental)

**定義**: 將系統分成多個獨立模組,逐步開發、測試、交付。

**流程**:
```
模組1: 分析→設計→實作→測試→交付 (核心功能)
模組2: 分析→設計→實作→測試→交付 (功能擴充)
模組3: 分析→設計→實作→測試→交付 (進階功能)
```

**特性**:
- ✅ 及早交付核心功能
- ✅ 風險分散
- ✅ 易於變更與擴充
- ❌ 需要良好模組化設計
- ❌ 整合測試複雜

### 四、軟體測試完整知識

#### (1) 黑箱測試 vs 白箱測試

| 項目 | 黑箱測試 (Black Box) | 白箱測試 (White Box) |
|------|---------------------|---------------------|
| **關注點** | **功能需求** | **程式結構** |
| **測試依據** | 需求規格書 | 程式碼、流程圖 |
| **測試者** | 不需了解程式內部 | **需了解程式邏輯** |
| **測試範圍** | 輸入→輸出 | **程式執行路徑、邏輯** |
| **測試方法** | 等價類劃分、邊界值分析 | **語句覆蓋、分支覆蓋、路徑覆蓋** |
| **優點** | 模擬使用者觀點、不需程式知識 | **深入測試邏輯、發現隱藏錯誤** |
| **缺點** | 無法檢查程式邏輯錯誤 | **成本高、需程式專業** |
| **範例** | 測試「關稅計算結果是否正確」 | **測試「計算流程的每個if分支」** |

**黑箱測試技術**:
1. **等價類劃分**: 將輸入分為有效/無效類別
   - 範例: 關稅稅率 (0-5%, 5-10%, >10%) 各選一個測試
2. **邊界值分析**: 測試邊界值
   - 範例: 稅率0%, 5%, 10% (邊界值)
3. **錯誤推測**: 根據經驗推測可能錯誤

**白箱測試技術**:
1. **語句覆蓋 (Statement Coverage)**: 每行程式碼至少執行一次
2. **分支覆蓋 (Branch Coverage)**: 每個if/else分支都測試
3. **路徑覆蓋 (Path Coverage)**: 所有可能執行路徑都測試

---

#### (2) 測試階段

```
單元測試 (Unit Testing)
├─ 測試最小程式單位 (函數、類別)
├─ 開發者自行測試
└─ 工具: JUnit, PyTest

        ↓

整合測試 (Integration Testing)
├─ 測試模組間的介面與互動
├─ 範例: API呼叫、資料庫連線
└─ 由下而上、由上而下、三明治測試

        ↓

系統測試 (System Testing)
├─ 測試完整系統是否符合需求
├─ 包含功能、效能、安全測試
└─ 測試團隊執行

        ↓

驗收測試 (Acceptance Testing)
├─ 確認系統符合客戶需求
├─ 使用者參與 (UAT)
└─ 最終決定是否上線
```

---

#### (3) 靜態分析 vs 動態分析

| 項目 | 靜態分析 (Static Analysis) | 動態分析 (Dynamic Analysis) |
|------|--------------------------|---------------------------|
| **執行程式** | **不執行** | **執行程式** |
| **分析對象** | 原始碼、編譯檔 | **執行時行為** |
| **時機** | 開發階段 | **測試階段** |
| **優點** | 及早發現問題、覆蓋率高 | **發現執行時錯誤、真實環境** |
| **缺點** | 可能誤報、不知實際影響 | **測試成本高、覆蓋率難達100%** |
| **工具** | SonarQube, Checkmarx | **OWASP ZAP, Burp Suite** |
| **檢測項目** | 語法錯誤、程式碼品質、安全漏洞 | **SQL Injection、XSS、記憶體洩漏** |

---

#### (4) 弱點掃描 vs 滲透測試

| 項目 | 弱點掃描 (Vulnerability Scanning) | 滲透測試 (Penetration Testing) |
|------|--------------------------------|-------------------------------|
| **目的** | **發現已知漏洞** | **模擬真實攻擊** |
| **方式** | 自動化工具掃描 | **人工測試** |
| **深度** | 廣而淺 | **深入攻擊** |
| **成本** | 低 | **高** |
| **頻率** | 定期 (每月/每季) | **年度** |
| **侵入性** | 非侵入式 (預設) | **侵入式 (模擬攻擊)** |
| **報告** | 清單式漏洞列表 | **攻擊路徑、風險評估、修補建議** |
| **工具** | Nessus, OpenVAS | **Metasploit, Kali Linux** |
| **適用** | 日常維運 | **重大系統上線前、年度稽核** |

## 【重點題目解析】114-112年

### 題目 1: 黑箱測試 vs 白箱測試 (114年關務三等)

#### 📖 原題

> **題目**: 假設你負責測試海關貨物申報與關稅計算系統,請設計一個黑箱測試案例與一個白箱測試案例,並說明其測試範圍與目標。(24分)

1. **黑箱測試案例**:定義 → 測試範圍 → 測試目標 → 具體案例

**一、黑箱測試案例**

**(1) 定義**

黑箱測試 (Black Box Testing) 是一種**不考慮程式內部結構**,只關注**輸入與輸出關係**的測試方法。測試者從**使用者角度**驗證系統功能是否符合需求規格,不需了解程式碼實作細節。

**(2) 測試範圍**

針對「海關貨物申報與關稅計算系統」,黑箱測試範圍包括:
- 貨物申報資料輸入介面
- 關稅計算結果輸出
- 系統回應時間
- 錯誤訊息顯示
- 多種貨物類別與稅率組合

**(3) 測試目標**

- ✅ 驗證系統**功能正確性** (關稅計算是否正確)
- ✅ 確認系統**符合需求規格書**
- ✅ 檢查**邊界值與異常輸入**的處理
- ✅ 模擬**真實使用者操作情境**

**(4) 具體測試案例設計**

**二、白箱測試案例**

**(1) 定義**

白箱測試 (White Box Testing) 是一種**深入程式內部結構**的測試方法,測試者需**了解程式碼邏輯**,驗證所有執行路徑、條件分支、迴圈是否正確運作。

**(2) 測試範圍**

針對「海關貨物申報與關稅計算系統」,白箱測試範圍包括:
- 關稅計算演算法的程式碼路徑
- 條件判斷邏輯 (if-else 分支)
- 迴圈執行 (多筆貨物計算)
- 例外處理邏輯
- 資料庫查詢語句

**(3) 測試目標**

- ✅ 確保**所有程式碼路徑至少執行一次** (語句覆蓋)
- ✅ 驗證**所有條件分支都被測試** (分支覆蓋)
- ✅ 檢查**邏輯錯誤與潛在漏洞**
- ✅ 提高**程式碼覆蓋率** (Code Coverage)

**(4) 具體測試案例設計**

**假設關稅計算程式邏輯** (範例):

```python
def calculate_customs_duty(price, hs_code, origin_country):
    # Step 1: 驗證輸入
    if price <= 0:  # 分支A
        raise ValueError("價格必須大於0")
    
    # Step 2: 查詢稅率表
    tax_rate = get_tax_rate(hs_code)
    
    # Step 3: 檢查優惠稅率 (FTA自由貿易協定)
    if origin_country in FTA_COUNTRIES:  # 分支B
        discount_rate = get_fta_discount(origin_country)
        tax_rate = tax_rate * (1 - discount_rate)
    
    # Step 4: 計算關稅
    duty = price * tax_rate
    
    # Step 5: 檢查免稅額度
    if duty < DUTY_FREE_THRESHOLD:  # 分支C
        duty = 0
    
    return duty
```

**白箱測試案例設計**:

**測試案例 1: 語句覆蓋測試 (Statement Coverage)**

| 測試項目 | 內容 |
|---------|------|
| **測試目的** | 確保每行程式碼至少執行一次 |
| **測試資料** | - 測試1: price=5000, hs_code="8471.30", origin_country="US"<br>- 測試2: price= -100 (觸發例外) |
| **測試路徑** | - 執行Step 1 → Step 2 → Step 3 → Step 4 → Step 5<br>- 執行Step 1 (分支A) → 拋出例外 |
| **覆蓋率** | 語句覆蓋率 = **100%** (所有程式碼都執行到) |

**測試案例 2: 分支覆蓋測試 (Branch Coverage)**

| 測試項目 | 內容 |
|---------|------|
| **測試目的** | 測試所有if-else分支 |
| **分支分析** | - 分支A: price \u003c= 0 (True/False)<br>- 分支B: 原產國在FTA清單 (True/False)<br>- 分支C: 關稅 \u003c 免稅額 (True/False) |
| **測試資料** | 測試1: price=2000, origin="TW" (非FTA), duty=100 (低於免稅額)<br>→ A:False, B:False, C:True<br><br>測試2: price=50000, origin="SG" (FTA), duty=2000 (高於免稅額)<br>→ A:False, B:True, C:False<br><br>測試3: price=-500<br>→ A:True |
| **覆蓋率** | 分支覆蓋率 = **100%** (所有分支都測試到) |

**測試案例 3: 路徑覆蓋測試 (Path Coverage)**

| 測試項目 | 內容 |
|---------|------|
| **測試目的** | 測試所有可能的執行路徑組合 |
| **路徑列舉** | 共 2×2×2 = **8條可能路徑**<br>1. A=F, B=F, C=F<br>2. A=F, B=F, C=T<br>3. A=F, B=T, C=F<br>4. A=F, B=T, C=T<br>5. A=T (後續不執行) |
| **測試資料** | 設計5組測試資料,覆蓋所有路徑 |
| **目標** | 路徑覆蓋率 = **100%** |

**測試案例 4: 程式碼品質檢查**

| 測試項目 | 內容 |
|---------|------|
| **測試目的** | 檢查程式碼安全性與品質 |
| **檢查項目** | - SQL Injection: `get_tax_rate(hs_code)` 是否驗證輸入?<br>- 例外處理: 是否捕捉資料庫連線失敗?<br>- 浮點數精度: 金額計算是否用Decimal避免誤差?<br>- 日誌記錄: 是否記錄計算過程供稽核? |
| **工具** | Static Code Analysis (SonarQube, Checkmarx) |

**測試覆蓋率類型**:

1. **語句覆蓋 (Statement Coverage)**: 每行程式碼至少執行一次

2. **分支覆蓋 (Branch Coverage)**: 每個if-else分支都測試

- **即時性**: 稅率可能隨政策調整,需定期更新
- **稽核性**: 所有計算須可追溯,保留日誌

**解答**:

**一、問題對應的SSDLC階段分析**

**(1) SSDLC五階段簡述**

安全系統開發生命週期 (SSDLC, Secure Software Development Life Cycle) 是將**資安整合至每個開發階段**的方法論，包含五大階段：

```
階段1: 需求分析 → 階段2: 設計 → 階段3: 實作 → 階段4: 驗證 → 階段5: 部署維護
(Requirements)   (Design)    (Implementation) (Verification) (Maintenance)
```

**SSDLC與傳統SDLC的關鍵差異**:
- 傳統SDLC: 資安在最後才考慮 (上線前才做弱點掃描)
- **SSDLC**: **每個階段都整合資安要求** (安全左移 Shift Left)

---

**(2) 問題1: 數據傳輸未使用標準加密協議**

**對應階段分析**:

此問題對應 **SSDLC 的多個階段缺失**:

**① 階段1: 需求分析階段 (Requirements) - 主要缺失**

**問題根源**:
- ❌ **未定義明確的安全需求**: 沒有規範「必須使用TLS 1.2以上加密協議」
- ❌ **未進行威脅建模**: 忽略「中間人攻擊 (Man-in-the-Middle)」風險
- ❌ **未考慮各國差異**: 假設所有國家都使用相同技術標準

**應有的作為**:
- ✅ 定義安全需求: 「所有跨境數據傳輸必須使用 **TLS 1.3** 加密」
- ✅ 威脅建模: 使用 **STRIDE** 分析傳輸過程威脅
  - Spoofing (偽冒): 假冒源頭國家海關
  - Tampering (竄改): 修改貨物申報資料
  - Information Disclosure (資訊洩漏): **攔截未加密傳輸** ← 本案問題
- ✅ 制定**最低安全基線 (Security Baseline)**:
  - 不接受低於TLS 1.2的連線
  - 不使用已知脆弱的加密演算法 (如RC4、DES)

**② 階段2: 設計階段 (Design) - 次要缺失**

**問題根源**:
- ❌ **架構設計未考慮向下相容風險**: 為了與舊系統相容,允許降級到不安全協議
- ❌ **未設計協議協商機制**: 沒有自動偵測並拒絕不安全連線

**應有的作為**:
- ✅ 設計**強制加密架構**:
  ```
  【各國海關】--TLS 1.3加密-→【API Gateway】--內部加密-→【雲端平台】
       ↓ 不支援TLS 1.3?
     【拒絕連線】+ 【通知管理員】+ 【提供技術協助】
  ```
- ✅ 實施**安全通道強制策略 (Enforce Secure Channel)**
- ✅ 設計**降級攻擊防護** (Downgrade Attack Protection)

**③ 階段4: 驗證階段 (Verification) - 測試未發現**

**問題根源**:
- ❌ **滲透測試未涵蓋**: 沒有測試「非標準加密協議」情境
- ❌ **相容性測試過於寬鬆**: 為求相容,允許不安全連線

**應有的作為**:
- ✅ **安全測試包含**:
  - SSL/TLS協議掃描 (使用 Nmap, SSLScan)
  - 中間人攻擊模擬測試
  - 降級攻擊測試 (嘗試強制使用TLS 1.0)

---

**(3) 問題2: 日誌紀錄未加密,可能被內部人員竊取**

**對應階段分析**:

此問題主要對應 **階段2 (設計) 與階段3 (實作)**:

**① 階段2: 設計階段 (Design) - 主要缺失**

**問題根源**:
- ❌ **未套用資料分類與保護設計**: 日誌含有敏感資料卻未加密
- ❌ **未實施最小權限原則**: 過多內部人員有權限存取日誌
- ❌ **未考慮內部威脅 (Insider Threat)**

**應有的作為**:
- ✅ **資料分類設計**:
  ```
  日誌內容分類:
  - 公開 (Public): 系統狀態、健康檢查 → 不需加密
  - 內部 (Internal): 存取記錄、操作日誌 → 加密存放
  - 機密 (Confidential): 含個資、貨物清單 → 加密 + 存取控制
  - 絕密 (Restricted): 金鑰、密碼 → 禁止記錄,或加密存放HSM
  ```
- ✅ **設計靜態資料加密 (Encryption at Rest)**:
  - 日誌寫入雲端儲存前,先用 **AES-256** 加密
  - 使用雲端服務商的 KMS (Key Management Service) 管理金鑰
- ✅ **存取控制設計 (RBAC)**:
  - 僅授權給「資安長」、「稽核人員」
  - 實施**職務分離 (Separation of Duties)**: 管理員不能是稽核員

**② 階段3: 實作階段 (Implementation) - 次要缺失**

**問題根源**:
- ❌ **開發人員未遵循安全編碼規範**: 使用雲端服務商預設設定 (通常未加密)
- ❌ **未檢查第三方服務的安全設定**

**應有的作為**:
- ✅ 實作時**強制開啟加密選項**:
  ```python
  # ❌ 錯誤: 使用預設設定 (未加密)
  s3.put_object(Bucket='logs', Key='access.log', Body=log_data)
  
  # ✅ 正確: 強制伺服器端加密 (SSE-KMS)
  s3.put_object(
      Bucket='logs',
      Key='access.log',
      Body=log_data,
      ServerSideEncryption='aws:kms',  # 啟用KMS加密
      SSEKMSKeyId='arn:aws:kms:...'     # 指定金鑰
  )
  ```
- ✅ **程式碼審查 (Code Review)** 檢查安全設定

**③ 階段4: 驗證階段 (Verification) - 測試未發現**

**問題根源**:
- ❌ **安全配置檢查不足**: 未驗證雲端儲存的加密設定

**應有的作為**:
- ✅ **雲端安全配置掃描**:
  - 使用工具: AWS Config, Azure Security Center, ScoutSuite
  - 檢查項目: S3 Bucket是否啟用加密、存取權限是否過寬
- ✅ **滲透測試情境**: 模擬「具內部權限的惡意員工」嘗試存取日誌

---

**問題對應總結**:

| 問題 | 主要缺失階段 | 次要缺失階段 | 根本原因 |
|------|------------|------------|---------|
| **問題1: 傳輸未加密** | **階段1 (需求)** | 階段2 (設計)、階段4 (驗證) | 未定義安全需求、未威脅建模 |
| **問題2: 日誌未加密** | **階段2 (設計)** | 階段3 (實作)、階段4 (驗證) | 未考慮內部威脅、未強制加密 |

**二、雲端數據安全處理與第三方供應商合規確認**

**(1) 雲端數據安全應在哪個SSDLC階段處理?**

**答案: 應在 SSDLC 的所有階段 都處理,但以下階段最關鍵**:

**① 階段1: 需求分析 (Requirements) - 定義安全要求**

- ✅ 定義雲端數據的**分類與保護等級**
- ✅ 明確**法規遵循要求**:
  - GDPR (歐盟個資保護法)
  - 台灣個資法
  - 關務保稅法規
- ✅ 制定**第三方供應商選擇標準**

**② 階段2: 設計 (Design) - 最重要階段**

- ✅ 設計**加密架構**:
  - 傳輸加密 (TLS 1.3)
  - 靜態加密 (AES-256)
  - 金鑰管理設計 (使用雲端KMS)
- ✅ 設計**多租戶隔離** (Multi-tenant Isolation)
- ✅ 設計**資料主權合規** (Data Residency):
  - 歐盟資料不得離開歐盟
  - 中國資料必須存放於中國境內

**③ 階段5: 部署維護 (Maintenance) - 持續監控**

- ✅ 監控雲端服務商的**合規狀態變化**
- ✅ 定期審查**存取日誌與異常行為**

---

**(2) 如何確保第三方雲端服務供應商符合安全標準?**

**方法一: 合約層面的要求 (Contractual Requirements)**

**① SLA (Service Level Agreement) 安全條款**

在雲端服務合約中,明確要求:

```markdown
第三方供應商必須符合以下安全標準:

【認證要求】
- ISO 27001 (資訊安全管理系統)
- ISO 27017 (雲端服務資訊安全)
- ISO 27018 (雲端個資保護)
- SOC 2 Type II (安全性、隱私性稽核報告)
- CSA STAR Level 2 (雲端安全聯盟認證)

【法規遵循】
- GDPR合規 (若處理歐盟資料)
- PCI DSS (若處理支付資料)
- HIPAA (若處理醫療資料)

【技術要求】
- 靜態資料加密: AES-256
- 傳輸加密: TLS 1.3
- 金鑰管理: 客戶自主管理 (BYOK - Bring Your Own Key)
- 日誌保留: 至少12個月
- 事件通報: 72小時內通知資安事件
```

**② DPA (Data Processing Agreement) 資料處理協議**

- 明確定義供應商為「資料處理者 (Data Processor)」,客戶為「資料控制者 (Data Controller)」
- 供應商**不得擅自使用客戶資料** (尤其不得用於AI訓練)
- 定義**資料刪除義務** (合約終止後30天內刪除所有資料)

**③ 賠償與罰則條款**

- 資料外洩: 每筆紀錄賠償 USD 100
- 服務中斷: 依停機時間比例退款
- 違反合規: 最高可解約並求償

---

**方法二: 技術層面的驗證 (Technical Verification)**

**① 供應商評估問卷 (Vendor Security Questionnaire)**

評估問項範例:

| 評估項目 | 問題 | 證據要求 |
|---------|------|---------|
| **資料加密** | 是否提供靜態資料加密? | 提供技術文件與配置截圖 |
| **存取控制** | 是否實施RBAC? | 提供IAM設定範例 |
| **事件應變** | 是否有24/7 SOC? | 提供事件應變計畫 (IRP) |
| **備份機制** | 資料備份頻率? | 提供RPO/RTO承諾 |
| **資料主權** | 資料存放位置? | 提供資料中心清單 |

**② 技術實測 (Technical Testing)**

**上線前測試**:
```
✅ 驗證加密設定:
   - 使用工具掃描儲存是否啟用加密 (aws s3api get-bucket-encryption)
   - 確認TLS版本 (openssl s_client -connect api.example.com:443)

✅ 存取控制測試:
   - 嘗試用未授權帳號存取 (應被拒絕)
   - 測試MFA強制執行

✅ 資料隔離測試:
   - 建立多個Tenant,確認資料不互通
   - 模擬跨租戶攻擊

✅ 日誌完整性測試:
   - 執行敏感操作,確認寫入日誌
   - 驗證日誌不可竄改 (使用CloudTrail Integrity)
```

**③ 滲透測試 (Penetration Testing)**

- 要求供應商提供**年度滲透測試報告** (由第三方執行)
- 或自行對雲端環境進行滲透測試 (需供應商授權)

---

**方法三: 持續監控與稽核 (Continuous Monitoring & Audit)**

**① 即時安全監控**

部署自動化監控工具:

```
【雲端安全態勢管理 (CSPM)】
- 工具: Prisma Cloud, Wiz, CloudGuard
- 功能:
  ✅ 自動偵測不安全配置 (如S3 Public Access)
  ✅ 監控合規狀態 (CIS Benchmark, NIST)
  ✅ 異常行為偵測 (如大量資料下載)

【日誌分析 (SIEM)】
- 整合雲端日誌至SIEM系統 (Splunk, ELK)
- 設定警報規則:
  ⚠️ 非預期地區的存取
  ⚠️ 大量資料匯出
  ⚠️ 權限異常變更
```

**② 定期稽核 (Regular Audit)**

**季度審查**:
- 檢視供應商的**SOC 2報告** (每年更新)
- 確認ISO 27001認證有效性

**年度稽核**:
- 派遣稽核團隊進行**現地稽核** (Site Audit)
- 要求供應商提供**滲透測試報告、漏洞掃描報告**
- 審查**事件應變演練記錄** (Incident Response Drill)

**③ 供應商持續評估 (Vendor Risk Management)**

```
每季評估供應商風險:
├─ 財務穩定性: 是否可能倒閉?
├─ 安全事件紀錄: 過去是否有資料外洩?
├─ 認證有效性: ISO證書是否到期?
└─ 服務品質: SLA達成率是否下降?

若評估不合格 → 啟動 Exit Strategy (退場策略)
```

---

**方法四: 架構層面的風險降低 (Architectural Risk Mitigation)**

**① 多雲策略 (Multi-Cloud)**

- 不要只依賴單一供應商 (避免 Vendor Lock-in)
- 關鍵資料**分散存放**:
  - 主要資料: AWS
  - 備份: Azure
  - 災難復原: Google Cloud

**② 資料加密金鑰自主管理 (BYOK/HYOK)**

```
【傳統雲端加密】
  雲端服務商管理金鑰 → 服務商有能力解密資料 ⚠️

【客戶自主金鑰 (BYOK)】
  客戶管理金鑰,上傳至雲端KMS → 服務商無法單獨解密 ✅

【客戶掌控金鑰 (HYOK)】
  金鑰存放於客戶本地HSM → 服務商完全無法解密 ✅✅
```

**③ 零信任架構 (Zero Trust)**

- 不因「在雲端內部」就信任
- 所有存取都需:
  - 身分驗證 (MFA)
  - 授權檢查 (RBAC)
  - 加密通道 (mTLS)

```
✅ 網路隔離: 不同VPC/vNet
✅ 存取控制: 不同帳號權限
✅ 資料隔離: 禁止生產資料流入開發環境
✅ 部署流程: Dev → Test → Prod (單向)
```

## 🎯 申論題答題技巧總結

### 技巧一:時間分配 (以25分題為例)

```
【建議時間分配】
📝 審題 (2分鐘):
   - 圈出關鍵字
   - 判斷題型 (比較題?應用題?)
   - 構思答題架構

✍️ 撰寫 (18分鐘):
   - 定義/背景 (3分鐘) → 5分
   - 核心內容 (10分鐘) → 15分
   - 總結/比較 (5分鐘) → 5分

🔍 檢查 (2分鐘):
   - 確認回答完整
   - 檢查錯別字
```

### 技巧二:答題架構範本

**比較題** (如:SSDLC vs SDLC、黑箱vs白箱):
```
1. 定義兩者 (各2-3行)
2. 比較表格 (5-7個面向)
3. 優缺點分析
4. 適用情境
```

**應用題** (如:選擇開發方法論):
```
1. 情境分析 (需求特性、專案特性)
2. 選擇結論 (明確說出選擇)
3. 選擇理由 (列舉3-5點優勢)
4. 潛在風險 (2-3點注意事項)
```

**流程題** (如:SSDLC五階段):
```
1. 整體流程圖
2. 逐階段詳述:
   - 目的
   - 工作內容 (3-5項)
   - 交付文件
3. 階段間的銜接關係
```

### 技巧三:加分要素

✅ **圖表化呈現**:
- 流程圖、表格、架構圖
- 視覺化幫助理解,容易得分

✅ **具體範例**:
- 不只理論,舉實際案例
- 範例:「如海關關稅計算系統...」

✅ **引用標準/工具**:
- OWASP、NIST、ISO 27001
- 具體工具:SonarQube、Jenkins

✅ **時事結合**:
- 雲端運算、零信任、AI整合  - 展現對產業動態的掌握

✅ **結構清晰**:
- 使用編號 (一、二、三)
- 使用小標題
- 重點用粗體或底線

### 技巧四:常見陷阱

❌ **只寫定義,沒有深入**:
- 例如只寫「瀑布式是順序開發」→ 不夠
- 應該寫:流程、特性、優缺點、適用情境

❌ **混淆概念**:
- SSDLC ≠ SDLC
- 黑箱測試 ≠ 單元測試
- 敏捷 ≠ DevOps

❌ **忽略題目情境**:
- 題目說「政府機關」→ 要提法規遵循
- 題目說「跨國」→ 要提資料主權

❌ **只寫優點,不寫缺點**:
- 任何方法都有trade-off
- 要平衡論述

### 技巧五:高分關鍵

**25分題目的配分參考**:

- 📌 **定義正確** (3-5分):必須精準
- 📌 **內容完整** (12-15分):涵蓋所有要點
- 📌 **結構清晰** (3-5分):易讀易懂
- 📌 **實務應用** (2-3分):展現實務經驗

**爭取滿分的策略**:
1. ✅ 答題完整性:要點都有提到
2. ✅ 專業術語正確:用業界標準用語
3. ✅ 邏輯連貫:前後呼應
