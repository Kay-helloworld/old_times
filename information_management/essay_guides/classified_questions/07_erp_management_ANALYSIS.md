## 📚 核心知識解析

### 一、商業智慧 (Business Intelligence, BI)

#### (1) BI定義

**商業智慧 (BI)**: 透過**蒐集、整合、分析、呈現**組織資料,將資料轉化為**可行動的洞察** (Actionable Insights),支援**商業決策**的技術、應用與實務。

**核心概念**:
```
資料 (Data)
  ↓ (整合、清理)
資訊 (Information)
  ↓ (分析、挖掘)
知識 (Knowledge)
  ↓ (應用)
智慧 (Intelligence) → 決策
```

**BI vs 傳統報表**:

| 面向 | 傳統報表 | 商業智慧 |
|------|---------|---------|
| **時效性** | 定期 (月報、週報) | **即時或準即時** |
| **資料範圍** | 單一來源 | **多來源整合** |
| **分析深度** | 簡單統計 | **深度分析、預測** |
| **互動性** | 靜態固定 | **動態探索** |
| **呈現方式** | 表格 | **互動式儀表板、視覺化** |
| **範例** | Excel報表 | Tableau, Power BI |

#### (2) BI三大活動與技術 (109年/112年關務必考!)

**活動1: 資料獲取 (Data Acquisition)**

**目的**: 從多個來源**萃取資料**,**轉換格式**,**載入**到資料倉儲

**技術: ETL (Extract, Transform, Load)**

```
【Extract (萃取)】
從多個來源系統抽取資料

來源:
├─ 交易系統 (OLTP): ERP, CRM
├─ 外部資料: 市場資料、社群媒體
├─ 非結構化: Email, 文件
└─ IoT: 感測器資料

抽取方式:
├─ 全量抽取 (Full Extraction): 首次載入
└─ 增量抽取 (Incremental): 僅抽取變更資料

【Transform (轉換)】
清理、轉換、整合資料

轉換作業:
├─ 資料清理 (Data Cleansing)
│   ├─ 處理缺失值 (Missing Values)
│   ├─ 去除重複 (Deduplication)
│   ├─ 修正錯誤 (Error Correction)
│   └─ 格式統一 (Standardization)
│
├─ 資料整合 (Data Integration)
│   ├─ 合併多來源資料
│   ├─ 解決衝突 (如同一客戶在不同系統有不同ID)
│   └─ 主資料管理 (MDM, Master Data Management)
│
├─ 資料轉換 (Data Transformation)
│   ├─ 格式轉換 (如日期: 2024/12/5 → 2024-12-05)
│   ├─ 單位轉換 (如幣別: USD → TWD)
│   └─ 編碼統一 (如性別: M/F vs 男/女 → 統一為M/F)
│
└─ 衍生欄位計算 (Derived Fields)
    ├─ 利潤 = 收入 - 成本
    ├─ 客戶年齡 = 當前年 - 出生年
    └─ RFM分數計算

【Load (載入)】
將轉換後資料載入資料倉儲

載入方式:
├─ 批次載入 (Batch Load): 定期(如每晚)
├─ 即時載入 (Real-time): 資料產生即載入
└─ 漸進載入 (Incremental): 僅載入新增/變更資料

【ETL工具】
├─ 商業工具: Informatica, Talend, IBM DataStage
├─ 開源工具: Apache NiFi, Pentaho
└─ 雲端服務: AWS Glue, Azure Data Factory
```

**範例: 零售業BI的ETL**

```
【Extract】
├─ POS系統: 銷售交易
├─ ERP系統: 庫存資料
├─ CRM系統: 客戶資料
└─ 網站: 線上瀏覽行為

【Transform】
統一客戶ID:
├─ POS: 會員卡號 "M12345"
├─ 網站: Email "user@example.com"
└─ 整合: 建立統一客戶主檔 "CUST_00001"

清理異常:
├─ 銷售金額 < 0 → 標記為退貨
├─ 年齡 > 120 → 修正或刪除
└─ 缺手機號碼 → 標記為待補

計算衍生欄位:
├─ 客戶生命週期價值 (LTV)
├─ 商品銷售排名
└─ 季節性指數

【Load】
載入到資料倉儲DW
→ 供OLAP分析使用
```

**活動2: 資料分析 (Data Analysis)**

**目的**: 對資料進行**多維度分析**,發現**模式、趨勢、異常**

**技術: OLAP (Online Analytical Processing, 線上分析處理)**

**OLAP定義**: 支援**複雜查詢**與**多維度分析**的技術,快速回應分析請求。

**多維度資料模型 (Multi-dimensional Data Model)**: 

```
【維度 (Dimension)】
分析的視角,如:
├─ 時間維度: 年、季、月、日
├─ 地區維度: 國家、縣市、區、門市
├─ 產品維度: 類別、品牌、型號
└─ 客戶維度: 年齡層、性別、會員等級

【度量 (Measure)】
要分析的數值,如:
├─ 銷售額
├─ 銷售量
├─ 利潤
└─ 客戶數

【多維度立方體 (OLAP Cube)】

        產品
         ↑
        /│
       / │
      /  │
     /   │
    /────┼───→ 時間
   │     │  /
   │     │ /
   │     │/
   └─────┘
      地區

每個格子 = 一個數值
範例: (2024年,台北,筆電) = $5,000,000銷售額
```

**OLAP操作**:

```
【切片 (Slice)】
固定一個維度,查看2維切面

範例: 
"2024年"各地區、各產品的銷售額
(時間維度固定為2024)

      產品
       ↑
       │
       │
       └────→ 地區

【切塊 (Dice)】
固定多個維度,查看子立方體

範例:
"2024 Q1"、"台北+新北"、"筆電類"
的銷售額

【下鑽 (Drill-down)】
從概要到明細

時間: 年 → 季 → 月 → 日
地區: 國 → 縣市 → 區
產品: 類別 → 品牌 → 型號

範例:
2024年銷售額 $100M
→ 下鑽到Q1: $30M
→ 再下鑽到1月: $12M

【上捲 (Roll-up)】
從明細到概要 (與下鑽相反)

日 → 月 → 季 → 年

【旋轉 (Pivot)】
改變維度在坐標軸的位置

原: 橫軸=時間, 縱軸=產品
旋轉後: 橫軸=產品, 縱軸=時間
```

**實例查詢**:

```
【查詢1: 切片+下鑽】
問: "2024年各地區筆電銷售如何?"
→ 切片: 固定2024年+筆電類
→ 得到: 台北$5M, 高雄$3M...

問: "台北的筆電銷售,各品牌如何?"
→ 下鑽: 從"筆電類"到"品牌"
→ 得到: Apple$2M, HP$1.5M...

【查詢2: 上捲+旋轉】
問: "每日銷售太細,看月度總計?"
→ 上捲: 從"日"到"月"

問: "改成橫向看產品,縱向看時間?"
→ 旋轉坐標軸
```

**OLAP vs OLTP比較** (重要!)

| 面向 | OLTP (交易處理) | OLAP (分析處理) |
|------|----------------|----------------|
| **全名** | Online **Transaction** Processing | Online **Analytical** Processing |
| **目的** | 日常營運、交易記錄 | **分析、決策支援** |
| **使用者** | 大量(全體員工) | 少量(管理者、分析師) |
| **資料範圍** | 當前資料 | **歷史資料**(數年) |
| **資料量** | GB-TB | **TB-PB** |
| **查詢** | 簡單、預定義 (如查訂單) | **複雜、臨機** (如多維度交叉分析) |
| **回應時間** | 毫秒-秒 (極快) | 秒-分鐘 |
| **操作類型** | **CRUD** (新增/讀取/更新/刪除) | 主要**讀取**(Read-only) |
| **更新頻率** | **持續** (Real-time) | 批次 (Batch, 如每晚ETL) |
| **資料庫設計** | 正規化 (3NF, 避免冗餘) | **非正規化** (星狀/雪花式, 優化查詢) |
| **範例系統** | ERP, CRM, 訂單系統 | **資料倉儲**, BI系統 |
| **範例查詢** | "查詢訂單編號12345的狀態" | "分析2020-2024各地區、各產品線的銷售趨勢" |

**為何要分開OLTP和OLAP?**

```
【若不分開】
❌ OLAP複雜查詢拖慢OLTP交易速度
❌ 交易系統無歷史資料(僅保留近期)
❌ 資料分散多系統,難以整合分析

【分開後】
✅ OLTP: 專注快速交易
✅ OLAP: 專注深度分析
✅ 透過ETL連接兩者
```

### 二、ERP企業資源規劃系統 (Enterprise Resource Planning)

#### (1) ERP定義

**ERP (Enterprise Resource Planning)**: 整合企業**所有核心業務流程**的資訊系統,透過**單一資料庫**實現**即時資訊共享**,提升營運效率與決策品質。

**核心特性**:

```
【整合性 (Integration)】
打破部門孤島
├─ 財務、生產、銷售、人資...全部整合
├─ 單一資料來源 (Single Source of Truth)
└─ 資料一致性

【即時性 (Real-time)】
交易發生→即時更新全系統
範例: 銷售下單→庫存立即扣減→財務自動記帳

【模組化 (Modular)】
可依需求選擇模組
├─ 小公司: 先導入財務+庫存
└─ 大公司: 導入全部模組

【最佳實務 (Best Practices)】
內建業界最佳流程
企業需調整流程配合系統 (BPR)
```

**ERP vs 傳統資訊系統**:

| 面向 | 傳統系統 | ERP系統 |
|------|---------|--------|
| **架構** | 各部門獨立系統 | **整合單一系統** |
| **資料庫** | 多個資料庫 | **單一資料庫** |
| **資料重複** | 高 (各系統都有客戶資料) | **無** (共用主資料) |
| **資料一致性** | 低 (可能不同步) | **高** (即時同步) |
| **跨部門流程** | 需人工轉檔、溝通 | **無縫整合** |
| **即時性** | 批次處理 (隔天才更新) | **即時** |
| **維護成本** | 高 (多套系統) | 中 (單一系統) |
| **範例** | 各部門用Excel + Access | SAP, Oracle ERP |

#### (3) BPR企業流程再造 (107年地特考題!)

**BPR定義**: **徹底重新思考**並**激進地重新設計**業務流程,以在**成本、品質、服務、速度**等關鍵指標上獲得**戲劇性改善**。

**為何ERP導入需要BPR?**

```
【傳統做法】
企業流程 → 客製化系統配合流程
問題: 流程可能低效、不最佳

【ERP導入】
ERP內建最佳實務流程
企業流程需調整配合ERP
→ 需要BPR!

【範例】
傳統: 採購單要15個人簽核 (層層上報)
ERP最佳實務: 依金額自動分級授權
  <10萬: 課長核准
  10-50萬: 經理核准
  >50萬: 總經理核准
→ BPR: 簡化簽核流程
```

---

**BPR四大要素**:

**要素1: 人員 (People)**

```
【改變】
├─ 角色重新定義
│   傳統: 採購人員只負責採購
│   BPR後: 採購人員負責供應商關係管理
│
├─ 技能提升
│   需培訓使用ERP
│
└─ 組織文化
    從「部門導向」→「流程導向」

【挑戰】
❌ 抗拒改變
❌ 擔心失業 (流程簡化→人力需求↓)
```

---

**要素2: 流程 (Process)**

```
【流程重新設計原則】
✅ 消除非增值活動
   傳統: 手工抄寫資料→重複輸入
   BPR: 自動化,一次輸入

✅ 簡化流程
   傳統: 採購流程15步驟
   BPR: 整合成5步驟

✅ 流程標準化
   不同分公司流程統一

✅ 授權下放
   減少不必要核准

【範例: 採購流程】
傳統:
1. 部門填紙本請購單
2. 送主管簽核
3. 採購人員手動輸入系統
4. 詢價、比價 (Excel)
5. 採購單簽核
6. 通知供應商 (傳真/電話)
7. 收貨、手動對帳
= 7-10天

BPR後 (ERP):
1. 部門線上請購 (庫存低自動觸發)
2. 系統自動選擇合格供應商
3. 電子採購單自動發送
4. 收貨掃描條碼→自動對帳
= 1-2天
```

---

**要素3: 技術 (Technology)**

```
【關鍵技術】
├─ ERP系統
│   └─ 整合平台
├─ 工作流程引擎 (Workflow)
│   └─ 自動化簽核路徑
├─ 條碼/RFID
│   └─ 自動化資料擷取
└─ 行動裝置
    └─ 隨時隨地存取

【技術賦能】
✅ 自動化手工作業
✅ 即時資訊
✅ 異常警示
```

---

**要素4: 策略 (Strategy)**

```
【BPR需與企業策略一致】

策略: 成本領導
→ BPR重點: 自動化、減少人力、精簡流程

策略: 差異化 (客戶服務)
→ BPR重點: 快速回應、客製化能力

策略: 快速擴張
→ BPR重點: 流程標準化、可複製性

【高階支持】
BPR是策略層級變革
需CEO全力支持!
```

#### (4) ERP導入挑戰 (107年地特考題!)

**為何ERP導入失敗率高?**

根據研究,約**50-70%的ERP專案**被視為失敗或未達預期。

**失敗原因**:

**原因1: 高階主管不支持**

```
【問題】
├─ 視ERP為IT專案,而非策略專案
├─ 不願投入時間參與
├─ 預算不足,中途減少投資
└─ 對BPR猶豫不決

【後果】
→ 缺乏資源
→ 抗拒改變無法解決
→ 專案停滯

【應對】
✅ 讓CEO擔任專案發起人 (Sponsor)
✅ 定期向董事會報告
✅ 列為公司年度KPI
```

---

**原因2: 低估複雜度與時程**

```
【問題】
├─ 以為「買套軟體,裝起來就好」
├─ 低估客製化需求
├─ 低估資料移轉難度
└─ 時程過於樂觀

【實際情況】
小型企業: 6-12個月
中型企業: 12-24個月
大型集團: 24-48個月!

【應對】
✅ 聘請有經驗的顧問
✅ 參考同業導入案例
✅ 分階段導入 (不要一次全上)
```

---

**原因3: 抗拒改變 (Change Resistance)**

```
【使用者抗拒】
❌ 「舊系統用習慣了,為何要改?」
❌ 「新系統太複雜」
❌ 「我的工作會被ERP取代嗎?」

【中階主管抗拒】
❌ 威脅到權力 (流程透明化)
❌ 跨部門整合→失去獨立性
❌ 績效KPI改變

【應對】
✅ 充分溝通: 說明Why (為何要變)
✅ 參與式設計: 讓使用者參與流程設計
✅ 教育訓練: 足夠的訓練時間
✅ 快速勝利 (Quick Wins): 先展示小成果
✅ 獎勵機制: 鼓勵配合改變
```

---

**原因4: 資料品質問題**

```
【問題】
舊系統資料:
├─ 不完整 (缺欄位)
├─ 不正確 (錯誤資料)
├─ 不一致 (格式不統一)
└─ 重複 (同一客戶多筆)

【後果】
垃圾進,垃圾出 (GIGO, Garbage In, Garbage Out)
→ ERP報表錯誤
→ 使用者失去信心

【應對】
✅ 上線前徹底清理資料
✅ 建立主資料管理 (MDM)
✅ 資料驗證規則
```

---

**原因5: 過度客製化**

```
【陷阱】
企業要求:
「ERP要100%符合現有流程」
→ 大量客製化

【後果】
❌ 成本暴增 (客製化很貴)
❌ 時程延誤
❌ 系統升級困難 (客製化部分可能不相容)
❌ 失去ERP最佳實務優勢

【應對】
✅ 80/20原則: 80%用標準功能, 20%客製化
✅ 優先改流程配合系統
✅ 真正無法改的才客製化
```

---

**為何需要ERP顧問?**

```
【顧問價值】

1. 專業知識
   ├─ 熟悉ERP功能
   ├─ 了解各行業最佳實務
   └─ 技術專業 (SAP, Oracle等)

2. 經驗累積
   ├─ 導入過數十個專案
   ├─ 知道常見陷阱
   └─ 有解決問題的方法論

3. 中立第三方
   ├─ 不受內部政治影響
   ├─ 敢於挑戰既有流程
   └─ 協調跨部門衝突

4. 加速導入
   ├─ 快速診斷問題
   ├─ 提供範本與工具
   └─ 縮短摸索時間

5. 降低風險
   ├─ 專案管理
   ├─ 品質把關
   └─ 確保如期上線

【顧問類型】
├─ ERP廠商顧問 (SAP, Oracle原廠)
├─ 系統整合商 (SI, 如資策會、精誠)
└─ 獨立顧問

【成本】
通常佔ERP總預算的 30-50%
但可大幅降低失敗風險!
```

### 三、CRM客戶關係管理 (Customer Relationship Management)

#### (1) CRM定義

**CRM**: 管理企業與**現有及潛在客戶**關係的**策略+流程+技術**,目標是提升**客戶滿意度、忠誠度、價值**,最終增加獲利。

**核心理念**:

```
【以客戶為中心】
傳統: 以產品為中心
CRM: 以客戶為中心

【客戶生命週期管理】
潛在客戶 → 新客戶 → 活躍客戶 → 忠誠客戶
                            ↓
                        (或)流失客戶

各階段不同策略:
├─ 潛在: 行銷吸引
├─ 新客: 優質服務,建立信任
├─ 活躍: 交叉銷售,向上銷售
├─ 忠誠: VIP服務,口碑推薦
└─ 流失: 挽回策略

【360度客戶視圖】
整合所有客戶接觸點:
├─ 銷售: 訂單歷史
├─ 行銷: 活動回應
├─ 客服: 問題記錄
├─ 網站: 瀏覽行為
└─ 社群: 互動記錄

→ 完整了解客戶
```

---

#### (2) CRM兩大類型 (104年完整考題!)

**類型1: 作業型CRM (Operational CRM)**

**定義**: 面向客戶的**前台系統**,自動化**銷售、行銷、客服**流程。

**三大支柱**:

```
【銷售自動化 (Sales Force Automation, SFA)】

功能:
├─ 潛在客戶管理 (Lead Management)
│   ├─ 潛在客戶來源追蹤
│   ├─ 資格評估 (Lead Scoring)
│   └─ 分派給業務員
│
├─ 機會管理 (Opportunity Management)
│   ├─ 商機追蹤
│   ├─ 銷售階段管理 (漏斗)
│   └─ 成交機率預測
│
├─ 報價與訂單管理
│   └─ 快速產生報價單
│
├─ 拜訪管理
│   └─ 行程安排、拜訪記錄
│
└─ 銷售預測
    └─ 根據管道預測營收

效益:
✅ 銷售流程標準化
✅ 商機不流失
```

**類型2: 分析型CRM (Analytical CRM)**

**定義**: 分析客戶資料,發掘**洞察**,支援決策。

**核心分析**:

```
【客戶分群 (Customer Segmentation)】

方法1: RFM分析
R (Recency): 最近一次購買時間
F (Frequency): 購買頻率
M (Monetary): 購買金額

範例:
客戶A: R=最近7天, F=每月1次, M=$5000
    → 高價值活躍客戶
客戶B: R=1年前, F=曾經每週, M=$500
    → 流失的一般客戶

行動:
高價值 → VIP服務, 專屬折扣
流失 → 挽回活動

方法2: 行為分群
網站瀏覽 → 僅瀏覽vs加入購物車vs結帳

方法3: 人口統計分群
年齡、性別、地區、職業
```

---

```
【客戶生命週期價值 (CLV, Customer Lifetime Value)】

公式:
CLV = Σ(每年利潤) / (1+折現率)^年數

簡化版:
CLV = 平均訂單金額 × 購買頻率 × 客戶壽命

範例:
客戶平均每次購買$1000
平均每年購買4次
平均當客戶5年

CLV = $1000 × 4 × 5 = $20,000

應用:
✅ 決定行銷預算 (CAC < CLV)
✅ 識別高價值客戶
✅ 預測未來營收

CAC (Customer Acquisition Cost):
獲取一個新客戶的成本

理想比例: LTV/CAC > 3
```

---

```
【流失預測 (Churn Prediction)】

使用機器學習預測客戶流失機率

特徵:
├─ 購買間隔拉長
├─ 客服抱怨增加
├─ 促銷活動不回應
└─ 互動頻率降低

模型:
邏輯迴歸、隨機森林、XGBoost

輸出:
客戶X流失機率: 75% (高風險)

行動:
主動聯繫、提供特別優惠、改善服務
```

```
【交叉銷售/向上銷售 (Cross-sell/Up-sell)】

交叉銷售: 推薦相關產品
範例: 買手機 → 推薦手機殼、保護貼

向上銷售: 推薦高階產品
範例: 考慮A方案 → 推薦B高階方案

技術: 推薦系統
├─ 協同過濾 (Collaborative Filtering)
│   「買A的人也買了B」
├─ 內容過濾 (Content-based)
│   根據產品相似度
└─ 關聯規則 (Association Rule)
    啤酒 → 尿布

範例:
Amazon: 「購買此商品的人也購買...」
效益: 客單價↑20-30%
```

### 五、其他管理系統與概念

#### (1) SCM供應鏈管理 (Supply Chain Management)

```
【定義】
管理從原料到最終客戶的**完整供應鏈**

【供應鏈流程】
供應商 → 製造商 → 配銷商 → 零售商 → 客戶
  ↑←───── 資訊流 ─────────────←↑
  →────── 物流 ──────────────→
  ←────── 金流 ─────────────←

【SCM系統功能】
✅ 需求預測
✅ 採購管理
✅ 生產計劃
✅ 庫存最佳化
✅ 物流配送
✅ 供應商協同

【長鞭效應 (Bullwhip Effect)】
定義: 需求資訊沿供應鏈向上游傳遞時,
     變異性逐漸放大

範例:
零售: 需求↑10%
配銷: 訂貨↑20% (怕缺貨)
製造: 生產↑30%
供應商: 原料↑40%

→ 上游過度反應

解法:
✅ 資訊共享 (VMI, Vendor Managed Inventory)
✅ 縮短前置時間
✅ 減少批量訂購
```

### 技巧五: 常見陷阱避免

**陷阱1**: 混淆 ERP vs CRM
- ❌ "CRM管理生產製造"
- ✅ 正確:
  - ERP = 企業**內部**流程整合
  - CRM = 企業與**客戶**關係管理

**陷阱2**: 混淆 OLTP vs OLAP
- ❌ "OLAP用於日常交易"
- ✅ 正確:
  - OLTP = **Transaction** (交易)
  - OLAP = **Analytical** (分析)

**陷阱3**: BI只是報表
- ❌ "BI就是產生報表"
- ✅ 正確: BI是完整流程 (獲取、分析、呈現) + 決策支援

**陷阱4**: BPR = IT專案
- ❌ "BPR是系統導入"
- ✅ 正確: BPR是**流程再造**,涉及人員/流程/技術/策略

**陷阱5**: 資料探勘=資料庫查詢
- ❌ "資料探勘就是SQL查詢"
- ✅ 正確: 資料探勘是**發現未知模式**,非查詢已知資訊
